# BIM 标书生成 App — 分阶段开发计划

本文档将整个 App 拆成可独立交付的阶段，每阶段有明确目标、交付物和验收标准，便于一步一步推进开发。

---

## 产品定位与设计原则

- **不做完全迁移**：现有 Dify 工作流（`标书生成wutu.yml`）仅作**参考**，新 App 以「功能更强、流程更清晰、可审核可干预」为目标。
- **允许并鼓励改进**：工作流中不合理或可优化的地方，在新 App 中应**重构或增强**，不必与原 YAML 行为一致。
- **各阶段表述**：文档中「参考 Dify / 与 YAML 对齐」处，均表示可先实现一版再迭代优化，验收以**业务效果**为准，而非与 Dify 输出逐字一致。

---

## 现有工作流可改进点（参考）

开发时可将下列问题作为优化方向，不必照搬现有实现：

| 方面 | 当前情况（YAML） | 可改进方向 |
|------|------------------|------------|
| 步骤与审核 | 一条线跑到底，无中间审核 | 每步可暂停、审核、重生成、加要点（见阶段 3/4） |
| 章节间延时 | 固定 10s × 3 个延时节点（约 30s/章） | 可配置或取消；或用队列限流替代固定 sleep |
| 检索与知识库 | 知识库 ID、API 写死在流程里 | 放入设置，支持多知识库或按任务选择 |
| 模型选择 | 分析用智谱、框架用 DeepSeek、正文用智谱等，硬编码 | 各步骤可选模型（设置中配置多个 Key，步骤可选 provider） |
| Prompt 与结构 | 大段 prompt 在节点内，难以版本化与 A/B | Prompt 存配置或 DB，支持多版本/标签，便于迭代 |
| 章节粒度 | 只能「全部」或「第 N 章」，无多选 | 支持勾选多章（如第 1、3、5 章）再生成 |
| 输出格式 | 仅 DOCX，模板固定 | 可选模板、可选仅导出 Markdown、后续可扩展 PDF 等 |

以上在分阶段实现时按需采纳，不必一次做完。

## 阶段总览

| 阶段 | 名称 | 目标 | 预估时间 |
|------|------|------|----------|
| 0 | 环境与骨架 | 项目能跑起来，前后端打通 | 1–2 周 |
| 1 | 文档上传与解析 | 上传招标文件并解析出文本 | 1–2 周 |

## 阶段 0：环境与骨架

**目标**：本地/开发环境可运行，前后端能互相调用，为后续功能打基础。

### 0.1 后端骨架

- [x] 创建项目目录，例如：`backend/`（FastAPI）、`backend/celery_app/`、`backend/tasks/`。
- [x] 配置 Python 虚拟环境，依赖：`fastapi`、`uvicorn`、`celery`、`redis`、`sqlalchemy`、`httpx`、`python-dotenv` 等。
- [x] 编写 `main.py` 或 `app/main.py`，启动 FastAPI，挂载一个健康检查接口，如 `GET /health` 返回 `{"status": "ok"}`。
- [x] 配置 Celery：连接 Redis 作为 broker/backend，写一个最简单的 Celery 任务（如 `tasks.demo.hello`）并能在本地执行成功。
- [x] 配置 PostgreSQL：用 SQLAlchemy 建一个空的 `engine` 和 `SessionLocal`，可先不建业务表，只保证能连上库。

**交付物**：  
- 后端可 `uvicorn app.main:app` 启动，访问 `/health` 有响应。  
- 执行 `celery -A app.celery worker` 后，调用 demo 任务能完成。

### 0.2 前端骨架

- [x] 使用 Vite + React + TypeScript 创建前端项目，例如：`frontend/`。
- [x] 安装 UI 库：Ant Design 或 Shadcn/ui，以及 TanStack Query、axios 或 fetch 封装。
- [x] 配置开发环境请求后端：如 `VITE_API_BASE=http://localhost:8000`，封装一个 `api.get('/health')` 并能在页面上显示后端返回结果。
- [x] 做一个最简布局：顶部导航 + 主内容区，主内容区里显示「BIM 标书生成」标题和健康检查结果。

**交付物**：  
- `npm run dev` 能启动前端，页面能请求后端并显示 `/health` 结果。

### 0.3 基础数据模型（可选在本阶段或阶段 1）

- [x] 设计并创建表：`tasks`（任务）、`task_steps`（步骤）。  
  - `tasks`：id, user_id（可先 nullable）, status, created_at, updated_at 等。  
  - `task_steps`：id, task_id, step_key（如 `upload` / `analyze` / `framework` / `chapters` / `export`）, status, input_snapshot, output_snapshot, error_message, created_at, updated_at。  
- [x] 提供 REST：`POST /api/tasks` 创建任务，`GET /api/tasks/{id}` 获取任务（含步骤列表）。  
  - 创建任务时可选初始化若干步骤，状态为 pending。

**验收标准**：  
- 能创建一条任务，能查到该任务及其步骤列表；前后端能配合调用上述接口。

## 阶段 1：文档上传与解析

**目标**：用户在前端上传招标文件（PDF/DOC/DOCX），后端落盘并解析出纯文本，写入任务步骤输出。

### 1.1 上传接口与存储

- [x] 后端：`POST /api/tasks/{task_id}/upload`，接收 multipart 文件，校验类型（pdf/doc/docx）与大小。
- [x] 将文件保存到本地目录或 MinIO/S3，路径或 object key 存入任务或步骤的 `input_snapshot`/附件表。
- [x] 步骤：创建或更新 `upload` 步骤，状态为 running → completed，并在 output 中记录文件路径或可下载 URL（仅内部使用，不暴露绝对路径）。

### 1.2 文档解析

- [x] 引入解析库：如 PyMuPDF（pdf）、python-docx（docx），必要时 doc 转 docx 用 LibreOffice 或 pandoc。
- [x] 实现 `parse_document(file_path) -> str`，返回纯文本；异常时返回错误信息。
- [x] 在 Celery 中实现任务：根据 `task_id` 取上传文件路径，调用 `parse_document`，将结果写入 `upload` 或新建 `extract` 步骤的 `output_snapshot`（如 `{"text": "..."}`）。

### 1.3 前端上传与状态

- [x] 任务创建后，进入「上传」步骤：提供文件选择 + 上传按钮，调用 `POST /api/tasks/{task_id}/upload`。
- [x] 上传成功后，触发「开始解析」（调用后端接口，由后端入队 Celery 解析任务）。
- [x] 轮询 `GET /api/tasks/{id}` 或 `GET /api/tasks/{id}/steps/upload`，解析完成后展示「解析完成，共 xxx 字」或预览前几百字。

**验收标准**：  
- 上传一个 PDF 或 DOCX 后，能在任务中看到解析出的文本内容（或字数），且无报错。

## 阶段 2：分析 + 框架生成

**目标**：用解析出的文本调用 LLM 做招标文件分析，并生成章节框架；API Key 可从设置读取（若阶段 0/1 未做设置，可先用环境变量）。

### 2.1 设置与 LLM 调用

- [x] 设计「设置」存储：表 `user_settings` 或 `api_config`（user_id/tenant_id, provider, api_key 加密, base_url 等），或先使用环境变量 `DEEPSEEK_API_KEY`、`ZHIPU_API_KEY`。
- [x] 封装 LLM 调用层：  
  - 输入：provider、api_key、model、messages、temperature 等。  
  - 请求 DeepSeek/智谱的 HTTP API（如 OpenAI 兼容格式），返回 content 或抛错。
- [x] 在 Celery 任务中：根据 task 的 user_id 查设置取 api_key（或 fallback 到 env），再调 LLM。

### 2.2 分析步骤

- [x] **参考** Dify 工作流中「深度分析招标文件」的 prompt，在本应用中实现分析步骤；prompt 放在常量或配置文件，便于后续优化（如拆分维度、调整篇幅）。
- [x] Celery 任务：读取 `extract` 步骤的 text，调用 LLM，将分析结果写入新步骤 `analyze` 的 output（如 `{"text": "..."}`）。
- [x] 提供 API：如 `POST /api/tasks/{id}/steps/analyze/run` 入队分析任务；前端轮询任务/步骤状态。

### 2.3 参数提取

- [x] **参考** Dify 中「参数提取器」的逻辑，用 LLM 或规则从分析结果中提取 `project_info`（JSON）、`bim_requirements`（数组）、`risk_points`（数组）；字段与结构可按业务需要增删或改名。
- [x] 结果写入步骤 `params` 的 output。

### 2.4 框架生成

- [x] **知识库检索（可插拔）**：  
  - 抽象为统一接口：按 query 调用「知识库」得到参考片段列表（如 `search(query, top_k) -> list[chunk]`），框架生成与按章生成只依赖该接口，不写死具体后端。  
  - 配置驱动：后端类型（如 `thinkdoc` / `ragflow`）、API 地址、API Key 等从环境变量或设置读取；切换知识库来源时「增加/切换适配器 + 改配置」即可，无需重写 2.4 业务逻辑。  
  - 首版可按 ThinkDoc（或 YAML 中的 doc.bluedigit.ai）实现一个适配器；后续若改用 RAGFlow 等，在本地/他机部署 RAGFlow，在应用里新增 RAGFlow 适配器并改配置即可。知识库的建库、加文档在 ThinkDoc/RAGFlow 侧完成，本应用只调其检索接口。
- [x] **参考**「生成文档框架」的 prompt 实现框架生成，输入：分析结果、bim_requirements、检索结果；输出：每行「第 X 章 标题」。可在此基础上优化章节类型与数量。
- [x] 解析框架文本为结构化列表 `[{number, title, full_name}]`，写入步骤 `framework` 的 output（如 `{"chapters": [...]}`）。

### 2.5 前端展示

- [x] 在任务详情中按步骤展示：解析 → 分析 → 参数 → 框架。  
- [x] 每步完成后展示对应结果：分析结果可折叠长文本；框架展示为章节列表。

**验收标准**：  
- 上传并解析后，能一键跑完「分析 → 参数 → 框架」，前端能看到章节框架列表；结构满足后续「按章生成」需要，质量可按实际招标文件抽样评估，不强求与 Dify 完全一致。

## 阶段 3：步骤化审核（框架层）

**目标**：框架生成后暂停，用户可查看框架；支持「重生成框架」「添加要点」后继续。  
（后续若需要，可将「等待用户审核」扩展到分析、参数等步骤，本阶段先做框架层。）

**思路（3.3 已按此实现）**：「添加要点」是对**当前框架的修改建议**，不是「带入按章生成」的补充说明。用户保存要点后，应点击「重新生成框架」，由 LLM 根据「当前框架 + 用户要点」更新框架（未提及的章节必须原样保留）；满意后再点「接受并继续」进入下一阶段。

### 3.1 步骤状态与分支

- [x] 明确步骤状态：pending / running / waiting_user / completed / failed。  
- [x] 框架步骤完成后置为 `waiting_user`，不再自动进入下一阶段；只有用户点击「接受并继续」或「重生成」或先「添加要点」再「重新生成」后再「接受并继续」后才变更。

### 3.2 重生成框架

- [x] 接口：`POST /api/tasks/{id}/steps/framework/regenerate`。  
  - 后端：将 framework 步骤重置为 running，重新入队「框架生成」Celery 任务；若有 `extra_points`，任务内只给 LLM「当前框架 + 用户要点」，不给招标分析/BIM/知识库，并按硬约束「未提及章节原样保留」；若无要点则照旧用 analyze、params 与检索结果完整生成。
- [x] 前端：在框架结果区域提供「重新生成框架」按钮，调用上述接口后轮询，直到 framework 再次 waiting_user 或 failed。

### 3.3 添加要点后继续

- [x] **保存要点**：接口 `POST /api/tasks/{id}/steps/framework/save-points`，body：`{ "added_points": ["要点1", "要点2"] }`。  
  - 仅当 framework 为 `waiting_user` 时允许：将 `added_points` 写入 framework 步骤的 `output_snapshot.extra_points`，**不改变步骤状态**。用户保存后应点击「重新生成框架」使框架按要点更新。
- [x] **接受并继续**：接口 `POST /api/tasks/{id}/steps/framework/accept`，body 可选：`{ "added_points": ["要点1", "要点2"] }`。  
  - 仅当 framework 为 `waiting_user` 时允许：若提供 `added_points` 则合并到 `extra_points`，然后将步骤状态置为 `completed`，表示用户已确认框架，可进入下一阶段。
- [x] 框架生成任务（run_framework）：若存在 `extra_points`，只给 LLM「当前框架 + 用户要点」，系统提示硬约束「未提及章节必须原样」；有要点时使用较低 temperature。若无要点则照旧完整生成。
- [x] 下游「按章生成」（阶段 4）是否将 `extra_points` 带入 prompt 由阶段 4 按需实现；本阶段重点为框架层「按要点更新框架」。

### 3.4 前端审核界面

- [x] 框架步骤为 `waiting_user` 时，展示章节列表 + 三个操作：「接受并继续」「重新生成框架」「添加要点」。  
- [x] 「添加要点」：弹出框输入多条要点，提交时调用 **save-points**（不调用 accept）；成功后提示「已保存要点，可点击『重新生成框架』使框架按您的要点更新」。  
- [x] 「接受并继续」：调用 **accept**，步骤变为 completed；页面在 completed 时显示「框架已确认，可进入按章生成。」，若有 extra_points 可展示「已添加 N 个要点」及可折叠列表。

**验收标准**：  
- 框架生成后可选择重生成或加要点；加要点后保存（save-points），再点「重新生成框架」能按要点更新框架且其他章节保持不变；点「接受并继续」后步骤为 completed，可进入下一阶段。详见各阶段指导中的《阶段3.3-添加要点后继续-操作与验证.md》。

## 阶段 4：按章生成与 DOCX 导出

**目标**：根据框架逐章生成内容，聚合为完整 Markdown，再转为 DOCX 供下载。

### 4.1 按章生成

- [x] **参考** Dify 中「章节内容生成」的流程，在本应用中实现按章生成：  
  - 对 framework.output.chapters 做过滤：支持「全部」「第 N 章」，**可选**支持多选（如第 1、3、5 章）。  
  - 对每一章顺序执行：小节大纲 LLM → 检索 → 章节正文 LLM；prompt 参考 YAML 并可按需优化；**延时**可配置或取消（不必照搬固定 10s×3）。  
  - 每章结果追加到任务级别的「章节内容列表」或写入步骤 `chapters` 的 output（如 `{"chapter_1": "...", "chapter_2": "..."}`）。
- [x] 步骤状态：可设 `chapters` 为 running，每完成一章更新一次进度（如 progress 字段或子步骤），便于前端展示「正在生成第 X 章」。

### 4.1.1 单章要点与重生成（在 4.1 稳定后做）

- [x] **思路**：与框架层「添加要点 → 重新生成」一致，对**已生成的单章**支持用户提交修改建议（要点），并支持「重新生成本章」仅重跑该章、按要点更新该章正文，其余章节不变。
- [x] **数据**：在 chapters 步骤的 output_snapshot 中支持 per-chapter 要点，例如 `chapter_points: { "1": ["要点A"], "3": ["要点B"] }`；或与现有 `chapters` 结构合并（如每章元数据中带 `extra_points`）。
- [x] **接口**：  
  - 保存某章要点：如 `POST /api/tasks/{id}/steps/chapters/save-points`，body `{ "chapter_number": 3, "added_points": ["要点1", "要点2"] }`，仅更新该章对应要点，不改变步骤状态。  
  - 重新生成本章：如 `POST /api/tasks/{id}/steps/chapters/regenerate`，body `{ "chapter_number": 3 }`（可选带 `added_points`），仅将第 3 章入队重生成；任务内读取该章当前正文与已存要点，构建「当前正文 + 用户要点」的 prompt（类似框架 extra_points），只写回该章，不覆盖其他章。
- [x] **前端**：在按章生成完成后的**章节预览**处，每章提供「添加要点」「重新生成本章」；保存要点后提示可点击「重新生成本章」使该章按要点更新。
- [x] **依赖与排期**：依赖 4.1（按章生成与预览已完成）；可与 4.2（聚合与模板）灵活安排——先做 4.2 再做 4.1.1 亦可。

**验收标准**：对已生成的某一章可保存要点；可触发「重新生成本章」且仅该章更新、其余章不变；前端在预览处可操作并看到更新结果。

### 4.2 聚合与模板

- [x] **参考** Dify 中「文档聚合」的模板结构，在本应用中实现：项目信息（project_info_md）+ 各章内容 + 风险点附录，拼成一份 Markdown；章节顺序与分隔符可按业务调整。

### 4.3 DOCX 导出

- [x] 使用 python-docx：将 Markdown 转成 Word（可先用简单规则：标题、段落、列表、表格），输出文件存到存储，并记录文件 URL 或路径。
- [x] 接口：`GET /api/tasks/{id}/download` 或 `GET /api/tasks/{id}/artifacts/docx` 返回文件流或重定向到临时 URL。

### 4.4 前端

- [x] 在「按章生成」过程中轮询任务/步骤，展示「正在生成第 X 章」和进度条（若后端提供 progress）。  
- [x] 全部完成后展示「生成完成」，提供「下载 Word」按钮，调用下载接口。

**验收标准**：  
- 选择「全部章节」或指定章节能完整跑通，并下载到 DOCX；内容完整、结构清晰、满足标书使用需求即可，不以与 Dify 输出逐字一致为准。

## 阶段 5：设置与体验打磨

**目标**：在 App 设置中配置大模型 API Key；优化进度展示与错误处理。

### 5.1 设置页与 API Key

- [x] 后端：  
  - `GET /api/settings/llm`：返回各 provider 的配置状态（如已配置/未配置）、脱敏 key（如 `sk-***xxx`）及 base_url，不返回明文。  
  - `POST /api/settings/llm`：body 如 `{ "provider": "deepseek", "api_key": "sk-xxx", "base_url": "可选" }`，校验后加密写入 DB；支持仅更新 base_url。  
  - `GET /api/settings/models`、`POST /api/settings/models`：默认模型与各步骤（分析/参数/框架/按章）模型覆盖；模型名对应供应商（model registry）。  
- [x] 前端：设置页中「大模型 API」区块，为 DeepSeek、智谱等提供 API Key 输入框（password）、可选 Base URL 和保存按钮；保存后显示「已配置」或脱敏 key；「模型配置」区块可选择默认模型及各步骤模型。  
- [x] 任务执行时：Celery 通过 `get_llm_for_step(step_key)` 从设置/DB 解析 (provider, model)，再读取 api_key 与 base_url；若缺失则步骤失败并提示「请在设置中配置 xxx API Key」。  
- **说明**：API Key 使用 Fernet 加密存 DB；未设置 `SETTINGS_SECRET_KEY` 时使用固定开发占位密钥，保证 FastAPI 与 Celery 加解密一致。详见 [阶段5.1-设置页与API Key-操作与验证.md](各阶段指导/阶段5.1-设置页与API Key-操作与验证.md)。

### 5.2 进度与错误

- [x] 后端：步骤失败时在 `task_steps` 中记录 `error_message`；可选在任务上维护 `progress`（0–100）供前端展示（当前进度由 chapters 步骤 output_snapshot 提供，4.4 已用其做进度条）。
- [x] 前端：步骤失败时展示错误信息及「重试」按钮（重新入队该步骤）；长任务显示进度条或「第 X 章 / 共 Y 章」（4.4 已实现按章进度条；5.2 补全了解析步骤的重试按钮）。详见 [阶段5.2-进度与错误-操作与验证.md](各阶段指导/阶段5.2-进度与错误-操作与验证.md)。

### 5.3 基础体验

- [x] 统一加载态、成功/失败提示（Ant Design Message/Notification）：首页创建任务成功/失败使用 message.success / message.error，与任务详情、设置页一致；各页已有 Spin/loading。  
- [x] 任务列表页：首页展示历史任务（任务 ID、状态、创建时间），Table 支持行点击与「进入」链接进入任务详情；加载态与空态（暂无任务）已处理。详见 [阶段5.3-基础体验-操作与验证.md](各阶段指导/阶段5.3-基础体验-操作与验证.md)。

**验收标准**：  
- 在设置中填写并保存 API Key 后，新建任务能正常调用 LLM；未配置时提示明确；步骤失败可重试且界面有反馈。

## 阶段 6：审核专家模块（后期）

**目标**：支持「修改前 / 修改后」对比，标红（删除）、标绿（新增）。

### 6.1 数据与接口

- [x] 支持保存文档的多个版本（如原始生成版、人工修改版），或至少两段文本：original、modified。  
  - 实现：前端可传两段文本调用 POST /api/compare；可选在重生成前快照至 `task_steps.output_snapshot_before_regenerate`，供 GET diff 使用。
- [x] 接口：`POST /api/compare` 或 `GET /api/tasks/{id}/versions/{v1}/{v2}/diff`，输入两段文本，返回结构化 diff（如列表：`{type: 'add'|'del', text: '...'}`）。  
  - 实现：`POST /api/compare` 请求体 `{ original, modified }`，响应 `{ diff: [{ type, text }, ...] }`；可选 `GET /api/tasks/{id}/steps/framework/diff`、`GET /api/tasks/{id}/steps/chapters/diff?chapter_number=N` 返回最近一次要点变更的 original、modified、diff。详见 [阶段6.1-数据与接口-操作与验证.md](各阶段指导/阶段6.1-数据与接口-操作与验证.md)。

### 6.2 Diff 逻辑

- [x] 使用 diff-match-patch 或类似库生成行级/块级 diff；若需语义块级，可再叠加 LLM 做分段对比。  
  - 实现：保持 6.1 的 difflib 行级 diff，未引入 diff-match-patch；LLM 语义块级留待后续。
- [x] 前端：根据 diff 结果渲染：删除标红、新增标绿；或集成 Monaco Diff Editor 等组件。  
  - 实现：新增 `DiffView` 组件（删除标红、新增标绿）；`api/compare.ts` 封装 postCompare、getFrameworkDiff、getChaptersDiff；对比页 `/compare` 两文本框 + 对比按钮 + DiffView 展示。Monaco Diff Editor 未集成。详见 [阶段6.2-Diff逻辑与前端-操作与验证.md](各阶段指导/阶段6.2-Diff逻辑与前端-操作与验证.md)。

### 6.3 前端审核视图

- [x] 提供「审核」Tab 或页面：选择任务与版本（或上传两段文本），展示左右对比或统一视图下的红绿高亮。  
  - 实现：在对比页 `/compare` 扩展「从任务查看对比」（任务下拉、框架/章节类型、章节号、加载对比 + DiffView）；在任务详情页框架步骤旁增加「查看框架对比」、每章旁增加「查看本章对比」按钮，点击后 Modal 内展示 diff（修改前/修改后可折叠 + 红绿高亮）。统一视图为主，未做左右对比。详见 [阶段6.3-前端审核视图-操作与验证.md](各阶段指导/阶段6.3-前端审核视图-操作与验证.md)。
- [x] 可选：支持在界面上直接编辑「修改后」再次对比。  
  - 实现：对比页「从任务加载」结果下方提供可编辑「修改后」文本框与「再次对比」按钮，调用 postCompare 刷新 DiffView；任务详情页弹窗未做编辑再次对比。

**验收标准**：  
- 输入两段文本能生成 diff，并在前端正确显示标红标绿。
