# 自动审查模块方案（与现有代码结构贴合）

## 一、目标与范围

- **目标**：在「按章生成」完成后，对每一章进行校审，产出**校审意见**（废标项、幻觉、AI 套路等）；用户接受部分/全部意见后，可**按章重新生成**，将意见作为该章的补充要点注入现有重生成流程。
- **范围**：新增步骤 `review`、Celery 任务、校审专用提示词、API；**前端将校审作为独立模块**（左侧导航「校审」、校审列表页、单任务校审视图），任务详情页仅保留审查步骤状态与「进入校审」入口；复用现有 `chapters` 的 `chapter_points` 与 `regenerate_chapter`，不新增重生成逻辑。

---

## 二、与现有结构的关系

### 2.1 任务步骤与数据流

当前步骤顺序（见 `frontend/src/theme/stepStatus.ts` 的 `TASK_STEP_ORDER` 与 `backend/app/schemas/task.py` 的 `DEFAULT_INITIAL_STEPS`）：

```
upload → extract → analyze → params → framework → chapters → export
```

审查步骤插入位置建议：

```
… → chapters → review → export
```

- **步骤 key**：`review`。
- **是否加入默认步骤**：建议**可选**。方案 A：不加入 `DEFAULT_INITIAL_STEPS`，在「按章生成」完成后由前端展示「开始审查」入口，首次触发时创建 `step_key=review` 的 TaskStep 并入队。方案 B：将 `"review"` 加入 `DEFAULT_INITIAL_STEPS`（在 `chapters` 与 `export` 之间），则新建任务即带审查步骤，前端步骤条需同步加入 `review`。

### 2.2 依赖步骤与输入来源

| 输入 | 来源 | 用途 |
|------|------|------|
| 招标分析摘要 | analyze 步骤 `output_snapshot["text"]` | 校审时对照招标要求，查漏、防幻觉 |
| 参数提取结果 | params 步骤 `output_snapshot` | 废标点、BIM 要求、评分项、风险点等 |
| 章节列表 | framework 步骤 `output_snapshot["chapters"]` | 章节标题与编号 |
| 章节正文 | chapters 步骤 `output_snapshot["chapters"][str(N)]` | 被审内容 |
| 知识库片段 | 现有 `app.knowledge_base.search` | 按章或按标题检索，作为依据/参考 |

与现有 `tasks/chapters.py`、`tasks/framework.py` 的读库方式一致：通过 `TaskStep` 按 `task_id` + `step_key` 读取上述步骤的 `output_snapshot`。

### 2.3 复用现有能力

- **单章重生成**：`tasks/chapters.regenerate_chapter(task_id, chapter_number)`，已支持「当前正文 + 补充要点」的 prompt（`build_chapter_regenerate_messages`），要点来自 `output_snapshot["chapter_points"][str(chapter_number)]`。
- **接受校审后的行为**：用户对某章「接受校审意见」→ 将选中的校审条目标题/描述作为**补充要点**写入 chapters 步骤的 `chapter_points` → 调用现有 `regenerate_chapter`，无需新重生成链路。
- **chapters 步骤的 chapter_points**：当前已在 `tasks/chapters.regenerate_chapter` 中读取，并在 `routers/tasks.py` 的 `save_chapter_points` 中写入；审查接受时只需对同一字段做「合并或覆盖」后保存并触发重生成。

## 三、数据与存储

### 3.1 TaskStep（沿用现有表）

- **step_key**：`"review"`。
- **status**：`pending` | `running` | `completed` | `failed`（无 `waiting_user`，审查结果直接写入后即为完成；是否采纳由用户在前端操作后走「接受 → 重生成」）。
- **input_snapshot**：可选，存章节列表或任务元信息，便于排查。
- **output_snapshot**：必存，JSON，建议结构如下：

```json
{
  "chapters": {
    "1": [
      { "type": "废标项", "description": "…", "quote": "原文引用（可选）" },
      { "type": "幻觉", "description": "…", "quote": "" },
      { "type": "套路", "description": "…", "quote": "" }
    ],
    "2": [ … ]
  }
}
```

- **type** 建议枚举：`废标项` | `幻觉` | `套路` | `建议`（可与提示词约定一致）。
- **error_message**：失败时与现有步骤一致。

### 3.2 校审单条结构（与提示词输出约定一致）

便于前端展示与「接受」时传参，单条建议：

- `type: string` — 类型。
- `description: string` — 问题描述或修改建议。
- `quote: string`（可选）— 章节内原文引用。

前端「接受」时可将 `description`（或 `description + quote` 拼接）作为「补充要点」传入后端，后端写入 `chapter_points` 并触发重生成。

## 四、后端实现要点

### 4.1 新增 Celery 任务：`tasks/review.py`

- **run_review(task_id: int)**  
  - 前置条件：chapters 步骤 `status == "completed"` 且 `output_snapshot` 存在；analyze、params、framework 已完成（与 chapters 一致）。  
  - 逻辑：  
    1. 读取 framework 的 chapters 列表、chapters 的 `output_snapshot["chapters"]`。  
    2. 读取 analyze 的 text、params 的 risk_points / bim_requirements / scoring_items 等（可截断长度，参考 `config.CHAPTER_OUTLINE_ANALYZE_MAX_LEN`）。  
    3. 对**每一章**：用章节标题或正文摘要做 KB 检索（`kb_search`，top_k 可 5–8）；组装校审 prompt（见下）；调用 LLM；解析返回为 `list[{type, description, quote}]`。  
    4. 将结果写入 review 步骤的 `output_snapshot`（格式见 3.1），`status = "completed"`；任一章失败可整步失败或该章记为一条错误说明，由实现选择。

- **可选**：`run_review_chapter(task_id, chapter_number)`，仅审单章并只更新 `output_snapshot["chapters"][str(chapter_number)]`，便于「重生成后仅重审该章」。

### 4.2 校审提示词：`app/prompts.py`

- 新增**校审专用** system / user 模板（与生成类 prompt 分离），例如：
  - **REVIEW_SYSTEM**：角色为「BIM 技术标校审专家」，说明输出格式（如 JSON 数组或约定好的 Markdown 列表），以及三类重点：废标项、幻觉、AI 套路（可加「建议优化」）。
  - **REVIEW_USER_TEMPLATE**：占位符包括：`{{#chapter_full_name#}}`、`{{#chapter_content#}}`、`{{#analyze_excerpt#}}`、`{{#params_risk_bim_scoring#}}`、`{{#kb_context#}}`。
- 实现 `build_review_messages(chapter_full_name, chapter_content, analyze_text, params_risk_bim_scoring, kb_context)`，返回 `[system_message, user_message]`。
- 实现 `parse_review_output(llm_text) -> list[dict]`：从 LLM 返回中解析出 `{type, description, quote}` 列表（若为 JSON 则直接解析；若为 Markdown 则按约定正则抽取）。

### 4.3 模型配置（与现有步骤一致）

- **PlatformLlmConfig**：若希望审查可单独选模型，需新增字段 `review_model`（迁移：`ALTER TABLE platform_llm_config ADD COLUMN review_model VARCHAR(64);`）。
- **settings_store**：`STEP_KEYS` 增加 `"review"`；`get_model_config` / `set_model_config` 增加 `review_model` 的读写。
- **llm_resolver**：`get_llm_for_step("review")` 返回 default 或 `review_model`，与 analyze/params/framework/chapters 一致。
- 若暂不单独选模型，可让 review 使用 `chapters` 或 default 模型，则无需改表与设置页。

### 4.4 API：`app/routers/tasks.py`

- **POST `/api/tasks/{task_id}/steps/review/run`**（202）  
  - 校验：chapters 已完成；若当前无 review 步骤则创建 `TaskStep(task_id, step_key="review", status="pending")` 并 flush。  
  - 若 status 已为 `running` 返回 202 与「审查已在进行中」。  
  - 否则置 `status="running"`，入队 `run_review.delay(task_id)`，返回 202。

- **POST `/api/tasks/{task_id}/steps/review/accept`**（200）  
  - Body 建议：`{ "chapter_number": int, "accepted_items": string[] }`，其中 `accepted_items` 为用户勾选的校审条目的描述文本（或 id/下标，由前端与后端约定）。  
  - 逻辑：读取 chapters 步骤的 `output_snapshot`，将 `accepted_items` 合并到 `chapter_points[str(chapter_number)]`（可覆盖或 append，建议覆盖为该章本次接受的列表）；写回 chapters 步骤；入队 `regenerate_chapter.delay(task_id, chapter_number)`。  
  - 返回 200 + 提示「已接受校审意见，该章已加入重生成队列」。

- 审查结果无需单独 GET 接口：任务详情 `GET /api/tasks/{task_id}` 已返回所有 steps，其中 `step_key=review` 的 `output_snapshot` 即按章校审结果；校审模块页面通过任务详情接口或任务列表 + 详情获取同一数据即可。

## 五、前端实现要点（校审为独立模块）

**产品定位**：生成就是生成，校审就是校审。校审在**独立模块**中完成（左侧导航「校审」+ 校审工作台），任务详情页只保留审查步骤状态与进入校审的入口，不承载完整校审 UI，以减轻任务详情页负担。

### 5.1 左侧导航与路由

- **AppLayout / 导航**：左侧增加「校审」入口（与首页、对比、设置并列），点击进入校审模块。
- **路由建议**：
  - `/review` — 校审列表页（可审查的任务列表）。
  - `/review/:taskId` 或 `/tasks/:taskId/review` — 单任务校审视图（该任务的按章校审结果、接受并重生成）。
- **App.tsx**：为上述路由注册对应页面组件。

### 5.2 校审列表页（`/review`）

- **数据来源**：调用现有 `GET /api/tasks` 获取任务列表，前端根据每个任务的 `steps` 过滤出「章节已生成完成」的任务（chapters 步骤 status=completed）；可进一步按 review 步骤状态分类（待审查 / 审查中 / 已完成 / 失败）。
- **展示**：表格或卡片列表，列可包含：任务 ID、创建时间、审查状态（无 review 步骤视为「待审查」、review.status 为 running/completed/failed 等）、操作「进入审查」。
- **操作**：「进入审查」跳转到单任务校审视图（`/review/:taskId` 或 `/tasks/:taskId/review`）。若该任务尚无 review 步骤或 review 为 pending/failed，在单任务校审视图中可发起「开始审查」。

### 5.3 单任务校审视图（`/review/:taskId` 或 `/tasks/:taskId/review`）

- **职责**：仅做校审相关操作与展示，不展示完整任务流水线。
- **当 chapters 已完成且（无 review 步骤或 review 为 pending/failed）时**：展示「开始审查」按钮，调用 `POST .../steps/review/run`，轮询该任务详情直至 review 步骤 completed 或 failed。
- **当 review 步骤 status=completed 且 output_snapshot 存在时**：解析 `output_snapshot.chapters`，按章展示校审结果区块（折叠或列表）；每章展示该章校审条目的 `type`（Tag）、`description`、`quote`（若有）；每章提供「接受并重生成」按钮。
- **接受并重生成**：点击后弹窗勾选要采纳的条目（或「全部接受」），确认后调用 `POST .../steps/review/accept`，body 为 `{ chapter_number, accepted_items }`；提示「该章已加入重生成队列」，可轮询任务详情以感知 chapters 步骤重跑与完成。
- **可选**：提供返回校审列表的入口（如面包屑或「返回校审列表」链接）。

### 5.4 任务详情页（轻量审查入口）

- **步骤条**：`stepStatus.ts` 中 `TASK_STEP_ORDER` 在 `chapters` 与 `export` 之间插入 `"review"`；`STEP_TITLES` 增加 `review: "审查"`。任务详情步骤条展示 review 步骤状态（pending/running/completed/failed），与现有步骤一致。
- **不在此页展开完整校审 UI**：不在此页做按章校审结果列表、勾选、接受并重生成等。
- **入口**：在 review 步骤区块或步骤条旁提供「进入校审」按钮，点击后跳转到单任务校审视图（`/review/:taskId` 或 `/tasks/:taskId/review`），在校审模块中完成全部校审操作。若 review 为 pending 且尚未触发过审查，也可在该视图中点击「开始审查」。

### 5.5 与现有「要点补充」的区分

- 现有「保存要点」+「重生成」是针对用户**手动输入的要点**（在任务详情按章生成区块）；审查是**系统生成的校审意见**，用户在校审模块中选择后转为要点再重生成。接口上仍用 `review/accept` 区分来源，便于后续统计或回溯。

## 六、配置与扩展

- **超时**：review 单章调用 LLM 可复用或略放宽 `config.LLM_TIMEOUT_READ`（校审内容可能较长）。
- **知识库**：与 chapters 一致，依赖现有 `KNOWLEDGE_BASE_TYPE` 与 RAGFlow/ThinkDoc 配置；无知识库时 `kb_context` 可为空字符串。
- **后续**：若需「仅重审某一章」，可增加 `POST .../steps/review/run?chapter_number=N` 或单独接口，后端只调 `run_review_chapter(task_id, N)` 并只更新该章在 `output_snapshot["chapters"]` 中的结果。

## 八、实施顺序建议

1. **prompts**：校审模板 + `build_review_messages` + `parse_review_output`。  
2. **tasks/review.py**：`run_review` 单章循环 + KB + LLM + 写 review 步骤 output。  
3. **routers**：`review/run`、`review/accept`（accept 内写 chapter_points + regenerate_chapter）。  
4. **前端**：步骤条与「进入校审」入口（任务详情）；校审模块骨架（左侧导航、路由、校审列表页、单任务校审视图页）。  
5. **前端**：单任务校审视图中完整实现「开始审查」、按章校审结果展示、勾选接受并重生成。  
6. **可选**：review_model 配置与设置页、单章重审接口。

按上述顺序可实现「章节完成后 → 在校审模块中触发审查 → 展示校审意见 → 接受并重生成」的完整闭环；校审为独立模块，任务详情页仅保留步骤状态与入口，与现有步骤、存储和重生成逻辑保持一致。
