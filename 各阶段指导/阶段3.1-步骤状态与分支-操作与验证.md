# 阶段 3.1 步骤状态与分支 — 操作与验证指南（新手向）

本文档面向新手，按「需要准备什么 → 一步步操作 → 如何验证」说明阶段 3.1 的完整流程。阶段 3.1 实现**步骤状态与分支**：明确步骤状态为 pending / running / waiting_user / completed / failed；框架生成**成功**后置为 **waiting_user**，流程暂停等待用户审核；前端在 waiting_user 时展示章节列表与「等待您审核框架」类提示，**不**显示「开始生成框架」按钮。本阶段**不实现**「接受并继续」「重新生成」「添加要点」三个操作按钮（见阶段 3.2/3.3/3.4）。

---

## 一、阶段 3.1 要达成什么

- **步骤状态**：明确五类状态 — **pending** / **running** / **waiting_user** / **completed** / **failed**（在代码或文档中说明即可）。
- **框架完成后置为 waiting_user**：框架生成 Celery 任务**成功**时，将 framework 步骤的 status 设为 **waiting_user**，不再设为 completed；只有用户后续点击「接受并继续」或「重生成」或「添加要点并继续」（阶段 3.2/3.3 实现）后才变更。
- **前端展示**：当 framework 步骤 status 为 **waiting_user** 时，与 completed 时一样解析并展示**章节列表**；并显示「框架已生成，等待您审核。」及「请选择：接受并继续、重新生成或添加要点。」；**不**显示「开始生成框架」按钮。
- **失败不变**：框架生成**失败**时，framework 步骤仍为 **failed**，前端仍显示错误信息与「重试」按钮。

**验收标准**（来自开发计划）：框架步骤完成后置为 waiting_user；在任务详情中能看到章节列表与等待审核提示，且不出现「开始生成框架」按钮。

## 三、环境准备

- **无新增依赖**：后端、前端沿用现有依赖，无需额外安装。
- **服务顺序**：需要 **3 个终端**：后端（uvicorn）、Celery Worker、前端（npm run dev）。详见 [开发前-服务启动顺序.md](开发前-服务启动顺序.md)。

## 四、操作步骤（按顺序做）

### 步骤 1：确认 Redis、PostgreSQL 已运行

- **Redis**：CMD 或 PowerShell 执行 `redis-cli ping`，看到 `PONG`。
- **PostgreSQL**：pgAdmin 能连上本机或服务里 PostgreSQL 为「正在运行」。

若未运行，按 [开发前-服务启动顺序.md](开发前-服务启动顺序.md) 的「二」先启动。

---

### 步骤 2：启动后端（终端 1）

在终端（PowerShell 或 CMD）中执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

看到 `Uvicorn running on http://0.0.0.0:8000` 和 `Application startup complete.` 即表示后端就绪。**此窗口不要关**。

---

### 步骤 3：启动 Celery Worker（终端 2）

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
celery -A celery_app worker -l info -P solo
```

看到 `celery@xxx ready.` 即表示 Worker 就绪。**此窗口不要关**。

---

### 步骤 4：启动前端（终端 3）

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\frontend
npm run dev
```

看到 `Local: http://localhost:5173/` 即表示前端就绪。**此窗口不要关**。

---

### 步骤 5：跑全流程到框架生成并确认 3.1 行为

1. 浏览器打开 <http://localhost:5173>，点击**「创建任务」**，进入任务详情页。
2. 按顺序完成：**上传** PDF/DOCX → **解析完成** → **开始分析** → **分析完成** → **开始参数提取** → **参数完成** → **开始生成框架**。
3. 等待「框架生成中，请稍候…」结束（**成功**时不会出现「框架生成完成。」，而会出现「框架已生成，等待您审核。」）。
4. **前端确认**：
   - 应出现**「框架已生成，等待您审核。」**及**「请选择：接受并继续、重新生成或添加要点。」**；
   - 应出现**可折叠的「章节框架」**，展开后为章节列表（第 1 章 xxx、第 2 章 xxx…）；
   - **不应**出现「开始生成框架」按钮。
5. **后端确认**：记下当前任务 ID（如 URL 中 `/tasks/15` 的 15），在浏览器新标签页打开 `http://localhost:8000/api/tasks/15`（替换为你的任务 ID），在返回的 JSON 的 **steps** 中找到 **step_key 为 "framework"** 的项，确认其 **status** 为 **"waiting_user"**（不是 "completed"）。
6. 或在任务详情页底部的**步骤列表**表格中，查看 framework 行的「状态」列，应为 **waiting_user**。

若框架生成**失败**（如未配置 API Key），则 framework 步骤 status 应为 **failed**，页面显示错误信息与「重试」按钮；行为与阶段 2 一致。

---

### 步骤 6：（可选）仅验证已有任务

若已有任务已跑完框架生成且成功，可直接打开该任务详情页，确认：主文案为「框架已生成，等待您审核。」、副文案为「请选择：接受并继续、重新生成或添加要点。」、章节列表正常展示、无「开始生成框架」按钮；并用 GET /api/tasks/{id} 确认 framework 的 status 为 waiting_user。

## 五、验证（阶段 3.1 验收）

按下面四项逐项检查，都通过即表示阶段 3.1 完成。

### 验证 1：框架生成成功后 framework 步骤 status 为 waiting_user

1. 在框架生成**成功**完成后，调用 **GET /api/tasks/{任务id}**（或浏览器打开 `http://localhost:8000/api/tasks/{id}`）。
2. 在返回的 **steps** 中找到 **step_key 为 "framework"** 的项。
3. 其 **status** 应为 **"waiting_user"**（不是 "completed"）。
4. 或在任务详情页底部的**步骤列表**表格中，framework 行的「状态」为 **waiting_user**。

---

### 验证 2：前端在 waiting_user 时展示章节列表与等待审核提示

1. 在任务详情页，当 framework 步骤为 **waiting_user** 时，主内容区「步骤 4：框架生成」区块应显示：
   - **「框架已生成，等待您审核。」**；
   - **「请选择：接受并继续、重新生成或添加要点。」**（灰色副文案）；
   - **可折叠的「章节框架」**，展开后为**章节列表**（第 N 章 标题），与 output_snapshot.chapters 一致。
2. 若 chapters 为空，可显示「框架已生成，但未解析到章节」类提示；状态仍为 waiting_user。

---

### 验证 3：waiting_user 时不显示「开始生成框架」按钮

1. 在 framework 为 **waiting_user** 时，「步骤 4：框架生成」区块内**不应**出现「开始生成框架」按钮。
2. 仅当 framework 为 **pending** 或 **failed** 时才显示「开始生成框架」；waiting_user 表示已生成、等待用户操作，故不重复显示触发按钮。

---

### 验证 4：框架生成失败时仍为 failed，前端有错误与重试

1. 可选：对未完成参数提取的任务尝试触发框架生成，或暂时去掉 API Key 后重启 Celery 再跑一次框架生成。
2. framework 步骤 **status** 应为 **"failed"**，**error_message** 有说明。
3. 前端应显示「框架生成失败」及错误描述，并有**「重试」**按钮。

## 六、常见问题

| 现象 | 可能原因 | 处理办法 |
|------|----------|----------|
| 框架生成成功后 GET 任务详情里 framework 仍是 completed | 后端未改为 waiting_user 或 Celery 未重启 | 确认 `backend\tasks\framework.py` 成功分支中为 `status = "waiting_user"`；修改后需重启 Celery Worker |
| 前端仍显示「框架生成完成。」且无「等待您审核」 | 前端未对 waiting_user 做展示分支 | 确认 `TaskDetailPage.tsx` 中 frameworkStepDone 包含 waiting_user，且 waiting_user 时主文案为「框架已生成，等待您审核。」 |
| waiting_user 时仍出现「开始生成框架」按钮 | showFrameworkTrigger 未排除 waiting_user | showFrameworkTrigger 条件应为 pending 或 failed，不包含 waiting_user |
| 步骤列表中 framework 状态显示为 waiting_user 但前端无章节 | 前端解析 chapters 仅判断 completed | 解析 frameworkChapters 时条件应为 status === 'completed' \|\| status === 'waiting_user'（即 frameworkStepDone） |
| 想验证「接受并继续」等按钮 | 属阶段 3.2/3.3/3.4 | 本阶段不实现这三个按钮；通过 3.1 后进入 3.2/3.3/3.4 再实现 |
