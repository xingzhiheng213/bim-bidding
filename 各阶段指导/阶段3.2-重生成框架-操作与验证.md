# 阶段 3.2 重生成框架 — 操作与验证指南（新手向）

本文档面向新手，按「需要准备什么 → 一步步操作 → 如何验证」说明阶段 3.2 的完整流程。阶段 3.2 实现**重生成框架**：后端提供 **POST /api/tasks/{id}/steps/framework/regenerate**，在 framework 为 **waiting_user** 时将步骤重置为 running 并重新入队框架生成 Celery 任务；前端在 waiting_user 时展示**「重新生成框架」**按钮，点击后调用该接口并依赖现有轮询，直至 framework 再次变为 waiting_user 或 failed。

---

## 一、阶段 3.2 要达成什么

- **后端接口**：**POST /api/tasks/{task_id}/steps/framework/regenerate**（返回 202）。仅当 framework 步骤 status 为 **waiting_user** 时允许调用：将 framework 置为 running、清空 error_message、入队 `run_framework`，使用最新 analyze、params 与检索结果重新生成；若 framework 为 running 则返回 202 且 message「框架生成已在进行中」；若为 pending/failed 则返回 400「仅当框架等待审核时可重新生成」。
- **前端**：在「步骤 4：框架生成」区块、当 framework 为 **waiting_user** 时，在提示文案旁展示**「重新生成框架」**按钮；点击后调用 regenerate 接口，成功提示「框架已重新入队」并刷新任务；页面在 framework 为 running 时约每 2 秒轮询，完成后再次展示「框架已生成，等待您审核。」或失败信息与重试。

**验收标准**（来自开发计划）：框架步骤为 waiting_user 时可调用 regenerate 重新入队；前端有「重新生成框架」按钮，调用后轮询直至 framework 再次 completed（本项目中为 waiting_user）或 failed。

## 四、操作步骤（按顺序做）

### 步骤 1：确认 Redis、PostgreSQL 已运行

- **Redis**：CMD 或 PowerShell 执行 `redis-cli ping`，看到 `PONG`。
- **PostgreSQL**：pgAdmin 能连上本机或服务里 PostgreSQL 为「正在运行」。

若未运行，按 [开发前-服务启动顺序.md](开发前-服务启动顺序.md) 的「二」先启动。

---

### 步骤 2：启动后端（终端 1）

在终端（PowerShell 或 CMD）中执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

看到 `Uvicorn running on http://0.0.0.0:8000` 和 `Application startup complete.` 即表示后端就绪。**此窗口不要关**。

---

### 步骤 3：启动 Celery Worker（终端 2）

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
celery -A celery_app worker -l info -P solo
```

看到 `celery@xxx ready.` 即表示 Worker 就绪。**此窗口不要关**。

---

### 步骤 4：启动前端（终端 3）

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\frontend
npm run dev
```

看到 `Local: http://localhost:5173/` 即表示前端就绪。**此窗口不要关**。

---

### 步骤 5：进入 waiting_user 并点击「重新生成框架」

1. 浏览器打开 <http://localhost:5173>，进入一个**框架已生成、处于等待审核**的任务详情页（或按 3.1 流程跑全流程到「框架已生成，等待您审核。」）。
2. 确认页面显示**「框架已生成，等待您审核。」**、**「请选择：接受并继续、重新生成或添加要点。」**及**「重新生成框架」**按钮。
3. 点击**「重新生成框架」**：
   - 应出现「框架已重新入队」或类似成功提示；
   - 随后区块顶部应出现**「框架生成中，请稍候…」**（framework 变为 running，轮询约每 2 秒刷新）；
   - 等待数十秒到一两分钟后，应再次出现**「框架已生成，等待您审核。」**及新的章节列表（或失败时出现错误与「重试」）。
4. **后端确认**：在点击「重新生成框架」后、生成完成前，用 GET /api/tasks/{id} 查看 framework 步骤，status 应先为 **running**，完成后变为 **waiting_user** 或 **failed**。

---

### 步骤 6：（可选）用接口验证 regenerate

1. 确保某任务 framework 为 **waiting_user**（记下任务 id）。
2. 用 Postman 或 curl 调用：**POST http://localhost:8000/api/tasks/{id}/steps/framework/regenerate**。
3. 应返回 **202**，body 含 `"message": "框架已重新入队"`、`"step_key": "framework"`。
4. 再调用 GET /api/tasks/{id}，framework 步骤 status 应为 **running**；稍后轮询或再请求，应变为 **waiting_user** 或 **failed**。
5. 对 framework 为 **pending** 或 **failed** 的任务调用同一接口，应返回 **400**，detail 如「仅当框架等待审核时可重新生成」。对 **running** 的任务调用，可返回 202 且 message「框架生成已在进行中」。

## 五、验证（阶段 3.2 验收）

按下面三项逐项检查，都通过即表示阶段 3.2 完成。

### 验证 1：waiting_user 时调用 regenerate 返回 202，步骤变为 running 再结束

1. 在 framework 步骤为 **waiting_user** 时，调用 **POST /api/tasks/{id}/steps/framework/regenerate**。
2. 应返回 **202**，body 含 `"message": "框架已重新入队"`、`"step_key": "framework"`。
3. 该任务的 framework 步骤应先变为 **running**，由 Celery 执行完成后变为 **waiting_user**（或 **failed**）；GET /api/tasks/{id} 或步骤列表可确认。

---

### 验证 2：非 waiting_user 时调用 regenerate 返回 400（或 running 时 202 已在进行中）

1. 对 framework 为 **pending** 或 **failed** 的任务调用 POST .../regenerate，应返回 **400**，detail 如「仅当框架等待审核时可重新生成」。
2. 若 framework 为 **running**（例如刚点击重新生成尚未完成），再次调用 regenerate，应返回 **202** 且 message「框架生成已在进行中」，且不重复入队。

---

### 验证 3：前端展示按钮且点击后轮询更新

1. 在任务详情页，当 framework 为 **waiting_user** 时，应显示**「重新生成框架」**按钮（与「请选择：接受并继续、重新生成或添加要点。」同区）。
2. 点击后请求成功、出现「框架已重新入队」提示；页面自动轮询，先显示「框架生成中，请稍候…」，完成后再次显示「框架已生成，等待您审核。」及章节列表（或失败时错误与重试）。

## 六、常见问题

| 现象 | 可能原因 | 处理办法 |
|------|----------|----------|
| 点击「重新生成框架」后报 400 | 当前 framework 不是 waiting_user（如已是 running） | 确认步骤列表或 GET 任务详情中 framework 的 status 为 waiting_user 后再点；若刚点过且未完成，会处于 running，需等本次完成后再点 |
| 点击后无「框架生成中」或一直不更新 | Celery Worker 未开或未加载 tasks.framework | 开终端在 backend 下执行 `celery -A celery_app worker -l info -P solo`；确认 `celery_app\__init__.py` 的 include 含 `tasks.framework` |
| POST regenerate 返回 404 | 任务 id 错误或路由未加载 | 确认 URL 为 `/api/tasks/{数字}/steps/framework/regenerate`；确认后端已启动且带 `--reload` 已加载新路由 |
| 前端无「重新生成框架」按钮 | 前端未在 waiting_user 时渲染该按钮 | 确认 `TaskDetailPage.tsx` 在 `frameworkStep?.status === 'waiting_user'` 的区块内渲染了按钮并调用了 `regenerateFrameworkStep` |
