# 办公环境 — 数据库迁移与启动指南

本文档说明**到办公室后**如何拉取项目、准备数据库（bim_bidding）并启动服务。适用于本机安装 Redis / PostgreSQL / 后端 / 前端的开发方式（未用 Docker 时）。

---

## 一、适用场景

| 场景 | 说明 | 数据库是否需要重建 |
|------|------|---------------------|
| **A. 同一台办公机** | 之前就在这台电脑上开发，明天只是拉取最新代码 | **不需要**。PostgreSQL 里的 bim_bidding 和数据都还在，直接启动即可。 |
| **B. 换了一台机子** | 办公室新电脑，或之前在家开发、明天在办公室用另一台 | **需要**在这台机子上准备 PostgreSQL 和空库 bim_bidding；表由后端首次启动时自动创建。数据不会自动跟过来。 |
| **C. 从家里带数据到办公室** | 希望办公室的数据库里有家里的任务数据 | 需要在家导出（pg_dump），到办公室导入（见下文「五」）。 |

---

## 二、前置条件

1. **已从 Git 拉取最新项目**（例如 `git pull`），项目路径如 `d:\标书工作流`。
2. **本机已安装**：Redis、PostgreSQL、Python（含 backend 虚拟环境）、Node（含 frontend 依赖）。若未安装，需先按团队环境文档安装。
3. **backend 环境**：已复制 `backend/.env.example` 为 `backend/.env`，并按本机情况修改（见下文各场景）。

---

## 三、场景 A：同一台办公机（之前就在这台开发）

**适用**：你之前在这台办公电脑上跑过项目，PostgreSQL 里已有 bim_bidding。

### 步骤

1. **拉取最新代码**
   ```powershell
   cd d:\标书工作流
   git pull
   ```

2. **确认 Redis、PostgreSQL 已运行**
   - Redis：PowerShell 执行 `redis-cli ping`，返回 `PONG` 即正常；未运行则在「服务」中启动 Redis。
   - PostgreSQL：在「服务」中查看 `postgresql-x64-xx` 是否为“正在运行”；或 pgAdmin 能连上 localhost:5432 即正常。

3. **确认 backend/.env 中的数据库连接**
   - 打开 `backend/.env`，确认 `DATABASE_URL` 指向本机且库名为 `bim_bidding`，例如：
     ```env
     DATABASE_URL=postgresql://postgres:你的密码@localhost:5432/bim_bidding
     ```
   - 若之前就在用这份 .env，通常无需改。

4. **按《开发前-服务启动顺序》启动服务**
   - 终端 1：后端 `cd backend` → 激活虚拟环境 → `uvicorn app.main:app --reload`
   - 终端 2：Celery `cd backend` → 激活虚拟环境 → `celery -A celery_app worker -l info -P solo`
   - 终端 3：前端 `cd frontend` → `npm run dev`
   - 浏览器打开前端（如 http://localhost:5173），应能看到之前的任务列表；数据库**无需重新创建**。

## 四、场景 B：办公室新机或换了一台机子（首次使用）

**适用**：在这台电脑上第一次跑项目，或之前从未在这台机子上建过 bim_bidding。

### 步骤

1. **拉取最新代码**
   ```powershell
   cd d:\标书工作流
   git pull
   ```

2. **安装/启动 PostgreSQL**
   - 若本机未安装 PostgreSQL：先安装（建议 14+ 或 16），并记下安装时设置的 postgres 用户密码。
   - 在「服务」中启动 PostgreSQL，或确认其已设为自动启动。

3. **创建空数据库 bim_bidding**
   - **方式一（推荐）**：用 pgAdmin 连接本机 PostgreSQL → 右键“数据库”→ 创建 → 数据库名填 `bim_bidding` → 确定。
   - **方式二**：命令行（在 PostgreSQL 的 bin 目录下或已加入 PATH 时）：
     ```powershell
     psql -U postgres -c "CREATE DATABASE bim_bidding;"
     ```
     若提示需输入密码，按安装时设置的 postgres 密码输入。

4. **配置 backend/.env**
   - 复制示例并编辑：
     ```powershell
     cd d:\标书工作流\backend
     copy .env.example .env
     ```
   - 编辑 `.env`，至少设置：
     - `REDIS_URL=redis://localhost:6379/0`（本机 Redis）
     - `DATABASE_URL=postgresql://postgres:你的密码@localhost:5432/bim_bidding`（用户名/密码/主机/端口/库名按本机实际修改）
     - `UPLOAD_DIR=data/uploads`（或其它你有写权限的目录）
   - 其它项（如 API Key、SETTINGS_SECRET_KEY）可先留空或按需填写；不设 SETTINGS_SECRET_KEY 时应用内仍可配置 API Key（使用开发占位密钥）。

5. **安装 Redis**（若本机未装）
   - 安装 Redis 并启动（或设为服务）；Celery 依赖 Redis。

6. **安装 Python 依赖并启动后端（自动建表）**
   ```powershell
   cd d:\标书工作流\backend
   .venv\Scripts\activate
   pip install -r requirements.txt
   uvicorn app.main:app --reload
   ```
   - 后端首次启动时会执行 `Base.metadata.create_all()`，在 bim_bidding 中**自动创建**所有表（任务、步骤、设置、导出格式等）；**无需**手动建表或跑迁移脚本。
   - 若控制台出现 “Database connection OK”“Tables created or already exist”，说明数据库和表已就绪。

7. **启动 Celery 与前端**
   - 新开终端 2：`cd backend` → 激活虚拟环境 → `celery -A celery_app worker -l info -P solo`
   - 新开终端 3：`cd frontend` → `npm install`（若未装依赖）→ `npm run dev`
   - 浏览器打开前端，此时任务列表应为空（新环境）；可新建任务验证流程。

## 五、可选：从家里把数据库带到办公室

**适用**：你在家里开发时产生了任务数据，希望办公室的 PostgreSQL 里也有这些数据。

### 在家（源环境）

1. 确保本机 PostgreSQL 已启动，且 bim_bidding 中有数据。
2. 导出数据库（在 PowerShell 或 CMD 中，按实际安装路径调整 `psql`/`pg_dump` 所在目录）：
   ```powershell
   pg_dump -U postgres -F c -f d:\bim_bidding_backup.dump bim_bidding
   ```
   或导出为 SQL 文本（便于跨版本）：
   ```powershell
   pg_dump -U postgres -f d:\bim_bidding_backup.sql bim_bidding
   ```
3. 将 `bim_bidding_backup.dump` 或 `bim_bidding_backup.sql` 拷贝到 U 盘或网盘，带到办公室。

### 在办公室（目标环境）

1. 在办公室电脑上**先**完成「场景 B」中的步骤 1～3（拉代码、安装/启动 PostgreSQL、**创建空库 bim_bidding**）。
2. 若导出的是 **.dump** 格式：
   ```powershell
   pg_restore -U postgres -d bim_bidding -c d:\bim_bidding_backup.dump
   ```
   `-c` 表示先删除已存在对象再恢复，若目标库为空可省略；若有报错可忽略“已存在”类提示。
3. 若导出的是 **.sql** 格式：
   ```powershell
   psql -U postgres -d bim_bidding -f d:\bim_bidding_backup.sql
   ```
4. 配置 `backend/.env` 中的 `DATABASE_URL` 指向办公室本机的 bim_bidding（同场景 B）。
5. **注意 SETTINGS_SECRET_KEY**：若家里和办公室的 `.env` 中 `SETTINGS_SECRET_KEY` 不一致，则应用内保存的 API Key（加密存在 DB 里）在办公室无法正确解密，需在办公室的应用「设置」页重新配置 API Key；若两边使用相同密钥，则无需重配。

## 六、验证

- **后端**：浏览器打开 http://localhost:8001/health，应返回 `{"status":"ok"}`。
- **API 文档**：http://localhost:8001/docs，可查看接口。
- **前端**：http://localhost:5173（或 vite 提示的端口），能打开首页、任务列表；场景 A 应能看到之前的任务，场景 B 为空或仅有新测数据。
- **数据库**：用 pgAdmin 连接 bim_bidding，可查看 `tasks`、`task_steps`、`export_format_settings` 等表是否存在且有数据（场景 A 或导入后应有数据）。

## 七、常见问题

| 问题 | 可能原因 | 处理 |
|------|----------|------|
| 后端启动报错 “connection refused” 或 “could not connect to server” | PostgreSQL 未启动或 DATABASE_URL 错误 | 检查「服务」中 PostgreSQL 是否运行；检查 `.env` 中主机、端口、用户名、密码、库名是否为 bim_bidding。 |
| 后端报错 “database bim_bidding does not exist” | 未创建库 | 按场景 B 用 pgAdmin 或 `psql -U postgres -c "CREATE DATABASE bim_bidding;"` 创建。 |
| 前端能打开但任务列表一直加载或报错 | 后端未启动或 CORS/端口不对 | 确认后端已启动、前端 .env 中 VITE_API_BASE 指向后端（如 http://localhost:8001）。 |
| 导入备份后应用内 API Key 显示异常或无法使用 | SETTINGS_SECRET_KEY 与导出环境不一致 | 在办公室的「设置」页重新保存一次 API Key；或两边使用相同的 SETTINGS_SECRET_KEY。 |
