# 阶段 2.3 参数提取 — 操作与验证指南（新手向）

本文档面向新手，按「需要准备什么 → 一步步操作 → 如何验证」说明阶段 2.3 的完整流程。阶段 2.3 实现**参数提取**：分析完成后，前端可点击「开始参数提取」；后端入队 Celery 任务，读取 analyze 步骤的分析文本，调用 LLM 提取结构化参数（project_info、bim_requirements、risk_points），将结果写入 **params** 步骤的 output_snapshot；前端轮询并分块展示「项目信息」「BIM 要求」「风险点」，失败时展示错误信息与「重试」按钮。

---

## 一、阶段 2.3 要达成什么

- 后端提供 **POST /api/tasks/{task_id}/steps/params/run**，在 analyze 步骤已完成且含分析文本的前提下入队参数提取任务，返回 202；若该任务尚无 params 步骤，接口会先创建再入队。
- **Celery Worker** 执行参数提取任务：读取 analyze 步骤的 output_snapshot["text"]，用 `app.prompts` 中的参数提取模板调用 LLM（默认 deepseek / deepseek-chat，temperature 0.1），解析返回的 JSON 并归一化为 project_info（对象）、bim_requirements（字符串数组）、risk_points（字符串数组），写入 params 步骤的 output_snapshot；失败时 status 为 failed，error_message 有说明。
- **前端**：分析完成后出现「开始参数提取」按钮（**即使尚未有 params 步骤也会显示**，点击后由后端创建并入队）；点击后显示「参数提取中，请稍候…」并每约 2 秒轮询；完成后展示可折叠的「项目信息」「BIM 要求」「风险点」三块；失败时展示错误信息与「重试」按钮。
- 验收时能在**网页上**或**接口**完成：创建任务 → 上传 → 解析完成 → 分析完成 → 点击「开始参数提取」（或 POST params/run）→ 轮询至 params 完成 → 看到三块结构化结果；失败时能看到错误信息并可重试。

**验收标准**（来自开发计划）：分析完成后，能点击「开始参数提取」；入队后前端轮询直至 params 完成或失败；params 完成后能看到 project_info、bim_requirements、risk_points 分块展示；失败时能看到错误信息并可重试。

## 三、环境准备

### 1. 确认 Celery 加载了 params 任务

打开 `backend\celery_app\__init__.py`，确认 `include` 中包含 `"tasks.params"`，例如：

```python
include=["tasks.demo", "tasks.extract", "tasks.analyze", "tasks.params"],
```

若没有 `"tasks.params"`，请加上并保存；**修改后必须重启 Celery Worker**，否则点击「开始参数提取」会 202 但任务一直不执行或报「未注册任务」。

### 2. 配置 DeepSeek API Key

与阶段 2.2 相同：在 `backend\.env` 中配置 `DEEPSEEK_API_KEY=sk-...`。若 2.2 已配置，无需再改。修改 .env 后若 Celery 已启动，需**重启 Celery Worker** 才能读到新环境变量。

可选：若要为参数提取单独指定模型，可在 .env 中设置：
- `PARAMS_LLM_PROVIDER=deepseek`（默认）
- `PARAMS_LLM_MODEL=deepseek-chat`（默认）

### 3. 服务顺序

与 2.2 相同，需要 **3 个终端**：后端（uvicorn）、Celery Worker、前端（npm run dev）。详见 [开发前-服务启动顺序.md](开发前-服务启动顺序.md)。

## 四、操作步骤（按顺序做）

### 步骤 1：确认 Redis、PostgreSQL 已运行

- **Redis**：CMD 或 PowerShell 执行 `redis-cli ping`，看到 `PONG`。
- **PostgreSQL**：pgAdmin 能连上本机或服务里 PostgreSQL 为「正在运行」。

若未运行，按 [开发前-服务启动顺序.md](开发前-服务启动顺序.md) 的「二」先启动。

---

### 步骤 2：启动后端（终端 1）

在终端（PowerShell 或 CMD）中执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

看到 `Uvicorn running on http://0.0.0.0:8000` 和 `Application startup complete.` 即表示后端就绪。**此窗口不要关**。

---

### 步骤 3：启动 Celery Worker（终端 2）

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
celery -A celery_app worker -l info -P solo
```

看到 `celery@xxx ready.` 即表示 Worker 就绪。**此窗口不要关**。  
若未做这一步，或 `include` 中无 `tasks.params`，「开始参数提取」会返回 202 但 params 步骤会一直 running 或报未注册任务。

---

### 步骤 4：启动前端（终端 3）

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\frontend
npm run dev
```

看到 `Local: http://localhost:5173/` 即表示前端就绪。**此窗口不要关**。

---

### 步骤 5：在网页上完成「上传 → 解析 → 分析 → 参数提取」

1. 浏览器打开 <http://localhost:5173>，点击**「创建任务」**，进入任务详情页。
2. **上传**：在「上传招标文件」区块选择本机一个 **PDF 或 DOCX**，点击**「上传」**，等待上传成功（会自动触发解析）。
3. **解析**：等待「解析完成，共 xxx 字」和预览出现。
4. **分析**：点击**「开始分析」**，等待「分析完成。」和可折叠的「分析结果」出现。
5. **参数提取**：
   - 分析完成后应出现**「开始参数提取」**按钮（**即使步骤列表里还没有 params 也会显示**，因为 params 步骤在首次点击时由后端创建）。
   - 点击**「开始参数提取」**。
   - 应出现「参数提取已入队」或类似提示，随后显示**「参数提取中，请稍候…」**（页面会约每 2 秒自动轮询）。
6. **等参数提取完成**（通常十几秒到几十秒）：
   - 应出现**「参数提取完成。」**和可折叠的三块：**「项目信息」**（键值对）、**「BIM 要求」**（列表）、**「风险点」**（列表）。
7. 页面底部**步骤列表**中会出现 **step_key 为 "params"**、**status 为 "completed"**（首次点击后该步骤才会出现）。

若参数提取失败（如 LLM 返回非 JSON 或未配置 API Key），会显示红色**「参数提取失败」**及错误信息，可点击**「重试」**再次触发。

---

### 步骤 6：（可选）用接口验收

若习惯用 curl 或 Postman，可按下面做（把 `任务id` 换成实际数字）：

1. 创建任务：`POST http://localhost:8000/api/tasks`，body `{}`，记下返回的 `id`。
2. 上传文件：`POST http://localhost:8000/api/tasks/{任务id}/upload`，form-data 字段 `file` 为 PDF/DOCX。
3. 触发解析：`POST http://localhost:8000/api/tasks/{任务id}/steps/extract/run`，返回 202。
4. 轮询 `GET http://localhost:8000/api/tasks/{任务id}`，直到 steps 中 extract 的 status 为 completed。
5. 触发分析：`POST http://localhost:8000/api/tasks/{任务id}/steps/analyze/run`，返回 202；轮询直到 analyze 的 status 为 completed。
6. 触发参数提取：`POST http://localhost:8000/api/tasks/{任务id}/steps/params/run`，应返回 **202**，body 含 `"message": "参数提取已入队"`、`"step_key": "params"`。
7. 继续轮询 `GET /api/tasks/{任务id}`，直到 steps 中 params 的 status 为 completed；检查该步骤的 output_snapshot 为 JSON，含 `project_info`（对象）、`bim_requirements`（数组）、`risk_points`（数组）。

## 五、验证（阶段 2.3 验收）

按下面四项逐项检查，都通过即表示阶段 2.3 完成。

### 验证 1：开始参数提取接口返回 202 且任务入队

1. 在 analyze 步骤已完成、且 output_snapshot 含有效 "text" 的前提下，调用 `POST /api/tasks/{task_id}/steps/params/run`。
2. 应返回 **202**，body 含 `"message": "参数提取已入队"`、`"step_key": "params"`。
3. 若 analyze 未完成或任务不存在，应返回 **400**「请先完成分析」或 **404**；Celery 终端应收到 `run_params` 任务（收到后若条件不满足，任务会把 params 置为 failed）。

### 验证 2：params 步骤完成且含三块结构化数据

1. 调用 `GET /api/tasks/{id}`（id 为已调用过 params/run 的任务 id）。
2. 在 **steps** 中找到 **step_key 为 "params"** 的项。
3. 其 **status** 为 **"completed"**，**output_snapshot** 为合法 JSON，包含 **project_info**（对象）、**bim_requirements**（字符串数组）、**risk_points**（字符串数组）。

### 验证 3：前端能触发、轮询并分块展示结果

1. 在任务详情页，分析完成后出现「开始参数提取」按钮（**即使当时步骤列表里还没有 params 步骤**）；点击后提示入队，并显示「参数提取中，请稍候…」。
2. 轮询若干次后（不刷新页面）自动变为「参数提取完成」或「参数提取失败」。
3. 参数提取完成时展示「参数提取完成。」和可折叠的「项目信息」「BIM 要求」「风险点」三块，内容与 output_snapshot 一致；失败时展示错误信息与「重试」按钮。

### 验证 4：失败时 error_message 有说明、可重试

1. 可选：对未完成分析的任务调用 POST params/run，或注释 .env 中的 DEEPSEEK_API_KEY 后重启 Celery 再点击「开始参数提取」。
2. params 步骤 **status** 为 **"failed"**，**error_message** 中有明确说明（如「请先完成分析」或 LLM/JSON 解析错误）。
3. 前端展示「参数提取失败」及错误描述，点击「重试」可再次调用 params/run。

## 六、常见问题

| 现象 | 可能原因 | 处理办法 |
|------|----------|----------|
| 分析完成后没有「开始参数提取」按钮 | 前端逻辑要求分析完成且（无 params 或 params 为 pending/failed）才显示；若之前改过逻辑误加了「必须有 params 步骤」则不会显示 | 确认 `frontend\src\pages\TaskDetailPage.tsx` 中 showParamsTrigger 为 `analyzeStep?.status === 'completed' && (!paramsStep \|\| paramsStep.status === 'pending' \|\| paramsStep.status === 'failed')`，这样无 params 时也会显示按钮 |
| 点击「开始参数提取」返回 202 但 params 一直 running 或无变化 | 未开 Celery Worker，或 Worker 未加载 tasks.params | 开一个终端在 backend 下执行 `celery -A celery_app worker -l info -P solo`；检查 `celery_app\__init__.py` 的 `include` 含 `"tasks.params"`，修改后重启 Worker |
| params 步骤 status 为 failed，error_message 提示未配置 API Key 或 JSON 解析错误 | .env 未配置 DEEPSEEK_API_KEY，或 LLM 返回了非 JSON/缺键 | 在 `backend\.env` 中设置 `DEEPSEEK_API_KEY=sk-...` 并重启 Celery；若为 JSON 解析错误，多为模型返回格式不符合要求，可查看 error_message 具体内容 |
| POST params/run 返回 400「请先完成分析」 | 该任务尚未完成分析或 analyze 无 text | 先对该任务完成上传、解析、分析（POST extract/run、analyze/run 并轮询至 analyze 为 completed），再调 params/run |
| 修改 prompts 或 params 代码后结果没变 | Worker 未重启 | Celery Worker 需 Ctrl+C 停掉后重新执行 `celery -A celery_app worker -l info -P solo` 才会加载新代码 |
| Celery 启动时报错或收到「unregistered task」 | include 缺少 tasks.params 或 tasks.params 有语法错误 | 确认 `celery_app\__init__.py` 中 `include` 含 `"tasks.params"`；检查 `tasks\params.py` 无语法错误，必要时 `python -c "from tasks.params import run_params"` 验证 |
