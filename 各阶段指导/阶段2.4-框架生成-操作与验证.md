# 阶段 2.4 框架生成 — 操作与验证指南（新手向）

本文档面向新手，按「需要准备什么 → 一步步操作 → 如何验证」说明阶段 2.4 的完整流程。阶段 2.4 实现**框架生成**：参数提取完成后，前端可点击「开始生成框架」；后端入队 Celery 任务，读取 analyze 与 params 步骤，调用知识库检索（可选）与 LLM，生成「第 X 章 标题」框架并解析为结构化章节列表，写入 **framework** 步骤的 output_snapshot；前端轮询并展示「章节框架」列表，失败时展示错误信息与「重试」按钮。

---

## 一、阶段 2.4 要达成什么

- 后端提供 **POST /api/tasks/{task_id}/steps/framework/run**，在 params 步骤已完成且含有效 output 的前提下入队框架生成任务，返回 202；若该任务尚无 framework 步骤，接口会先创建再入队。
- **Celery Worker** 执行框架任务：读取 analyze 步骤的 text、params 步骤的 bim_requirements 等，调用知识库 `search(query, top_k)`（若已配置 ThinkDoc）；用 `app.prompts` 中的框架模板调用 LLM（默认 deepseek / deepseek-chat，temperature 0.4），解析返回的「第X章 标题」为 `[{number, title, full_name}]`，写入 framework 步骤的 output_snapshot 为 `{"chapters": [...]}`；失败时 status 为 failed，error_message 有说明。
- **前端**：参数提取完成后出现「开始生成框架」按钮（无 framework 步骤或 framework 为 pending/failed 时显示）；点击后显示「框架生成中，请稍候…」并每约 2 秒轮询；完成后展示可折叠的「章节框架」列表（第 N 章 标题）；失败时展示错误信息与「重试」按钮。
- **知识库（可选）**：若在 `.env` 中配置了 THINKDOC_API_KEY、THINKDOC_KB_IDS（及可选 KNOWLEDGE_BASE_TYPE=thinkdoc），框架生成时会调用 ThinkDoc 检索作为「知识库参考」；未配置时检索为空，框架仍可生成。

**验收标准**（来自开发计划）：上传并解析后，能一键跑完「分析 → 参数 → 框架」，前端能看到章节框架列表；结构满足后续「按章生成」需要。

## 三、环境准备

### 1. 确认 Celery 加载了 framework 任务

打开 `backend\celery_app\__init__.py`，确认 `include` 中包含 `"tasks.framework"`，例如：

```python
include=["tasks.demo", "tasks.extract", "tasks.analyze", "tasks.params", "tasks.framework"],
```

若没有 `"tasks.framework"`，请加上并保存；**修改后必须重启 Celery Worker**，否则点击「开始生成框架」会 202 但任务一直不执行或报「未注册任务」。

### 2. 配置 DeepSeek API Key

与阶段 2.2、2.3 相同：在 `backend\.env` 中配置 `DEEPSEEK_API_KEY=sk-...`。若已配置，无需再改。修改 .env 后若 Celery 已启动，需**重启 Celery Worker** 才能读到新环境变量。

可选：若要为框架生成单独指定模型，可在 .env 中设置：
- `FRAMEWORK_LLM_PROVIDER=deepseek`（默认）
- `FRAMEWORK_LLM_MODEL=deepseek-chat`（默认）

### 3. 知识库（可选）

若希望框架生成时使用 ThinkDoc 知识库检索作为「知识库参考」：

1. 在 `backend\.env` 中配置：
   - `THINKDOC_API_URL=https://doc.bluedigit.ai`（可选，有默认值）
   - `THINKDOC_API_KEY=td-你的密钥`
   - `THINKDOC_KB_IDS=知识库ID`（单个或多个逗号分隔）
2. 若已配置上述两项，`KNOWLEDGE_BASE_TYPE` 会自动推断为 `thinkdoc`；也可显式设置 `KNOWLEDGE_BASE_TYPE=thinkdoc`。
3. **验证检索**：在 backend 目录下、已激活 .venv 时执行 `python test_kb.py`，应看到「Chunks returned: N」且 N>0 及若干预览，表示 ThinkDoc 检索正常。

若不配置知识库，框架生成仍可运行，仅「知识库参考」为空。

### 4. 服务顺序

与 2.2、2.3 相同，需要 **3 个终端**：后端（uvicorn）、Celery Worker、前端（npm run dev）。详见 [开发前-服务启动顺序.md](开发前-服务启动顺序.md)。

## 四、操作步骤（按顺序做）

### 步骤 1：确认 Redis、PostgreSQL 已运行

- **Redis**：CMD 或 PowerShell 执行 `redis-cli ping`，看到 `PONG`。
- **PostgreSQL**：pgAdmin 能连上本机或服务里 PostgreSQL 为「正在运行」。

若未运行，按 [开发前-服务启动顺序.md](开发前-服务启动顺序.md) 的「二」先启动。

---

### 步骤 2：启动后端（终端 1）

在终端（PowerShell 或 CMD）中执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

看到 `Uvicorn running on http://0.0.0.0:8000` 和 `Application startup complete.` 即表示后端就绪。**此窗口不要关**。

---

### 步骤 3：启动 Celery Worker（终端 2）

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
celery -A celery_app worker -l info -P solo
```

看到 `celery@xxx ready.` 即表示 Worker 就绪。**此窗口不要关**。  
若未做这一步，或 `include` 中无 `tasks.framework`，「开始生成框架」会返回 202 但 framework 步骤会一直 running 或报未注册任务。

---

### 步骤 4：启动前端（终端 3）

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\frontend
npm run dev
```

看到 `Local: http://localhost:5173/` 即表示前端就绪。**此窗口不要关**。

---

### 步骤 5：在网页上完成「上传 → 解析 → 分析 → 参数提取 → 框架生成」

1. 浏览器打开 <http://localhost:5173>，点击**「创建任务」**，进入任务详情页。
2. **上传**：在「上传招标文件」区块选择本机一个 **PDF 或 DOCX**，点击**「上传」**，等待上传成功（会自动触发解析）。
3. **解析**：等待「解析完成，共 xxx 字」和预览出现。
4. **分析**：点击**「开始分析」**，等待「分析完成。」和可折叠的「分析结果」出现。
5. **参数提取**：点击**「开始参数提取」**，等待「参数提取完成。」和「项目信息」「BIM 要求」「风险点」三块出现。
6. **框架生成**：
   - 参数提取完成后应出现**「开始生成框架」**按钮（无 framework 步骤或 framework 为 pending/failed 时显示）。
   - 点击**「开始生成框架」**。
   - 应出现「框架已入队」或类似提示，随后显示**「框架生成中，请稍候…」**（页面会约每 2 秒自动轮询）。
7. **等框架生成完成**（通常几十秒到一两分钟，视文档与网络而定）：
   - 应出现**「框架生成完成。」**和可折叠的**「章节框架」**，展开后为章节列表（第 1 章 xxx、第 2 章 xxx…）。
8. 页面底部**步骤列表**中应出现 **step_key 为 "framework"**、**status 为 "completed"**。

若框架生成失败（如未配置 API Key 或 LLM 异常），会显示红色**「框架生成失败」**及错误信息，可点击**「重试」**再次触发。

---

### 步骤 6：（可选）用接口验收

若习惯用 curl 或 Postman，可按下面做（把 `任务id` 换成实际数字）：

1. 创建任务：`POST http://localhost:8000/api/tasks`，body `{}`，记下返回的 `id`。
2. 上传文件：`POST http://localhost:8000/api/tasks/{任务id}/upload`，form-data 字段 `file` 为 PDF/DOCX。
3. 触发解析：`POST http://localhost:8000/api/tasks/{任务id}/steps/extract/run`，返回 202；轮询直到 extract 的 status 为 completed。
4. 触发分析：`POST http://localhost:8000/api/tasks/{任务id}/steps/analyze/run`，返回 202；轮询直到 analyze 的 status 为 completed。
5. 触发参数提取：`POST http://localhost:8000/api/tasks/{任务id}/steps/params/run`，返回 202；轮询直到 params 的 status 为 completed。
6. 触发框架生成：`POST http://localhost:8000/api/tasks/{任务id}/steps/framework/run`，应返回 **202**，body 含 `"message": "框架已入队"`、`"step_key": "framework"`。
7. 继续轮询 `GET /api/tasks/{任务id}`，直到 steps 中 framework 的 status 为 completed；检查该步骤的 output_snapshot 为 JSON，含 **"chapters"** 数组，每项含 `number`、`title`、`full_name`。

## 五、验证（阶段 2.4 验收）

按下面四项逐项检查，都通过即表示阶段 2.4 完成。

### 验证 1：开始框架生成接口返回 202 且任务入队

1. 在 params 步骤已完成、且 output_snapshot 含有效 bim_requirements 等的前提下，调用 `POST /api/tasks/{task_id}/steps/framework/run`。
2. 应返回 **202**，body 含 `"message": "框架已入队"`、`"step_key": "framework"`。
3. 若 params 未完成或任务不存在，应返回 **400**「请先完成参数提取」或 **404**；Celery 终端应收到 `run_framework` 任务（收到后若条件不满足，任务会把 framework 置为 failed）。

### 验证 2：framework 步骤完成且含 chapters

1. 调用 `GET /api/tasks/{id}`（id 为已调用过 framework/run 的任务 id）。
2. 在 **steps** 中找到 **step_key 为 "framework"** 的项。
3. 其 **status** 为 **"completed"**，**output_snapshot** 为合法 JSON，包含 **"chapters"** 数组，每项含 **number**（数字）、**title**（字符串）、**full_name**（字符串，如「第1章 项目理解与分析」）。

### 验证 3：前端能触发、轮询并展示章节列表

1. 在任务详情页，参数提取完成后出现「开始生成框架」按钮（无 framework 或 framework 为 pending/failed 时）；点击后提示入队，并显示「框架生成中，请稍候…」。
2. 轮询若干次后（不刷新页面）自动变为「框架生成完成」或「框架生成失败」。
3. 框架生成完成时展示「框架生成完成。」和可折叠的「章节框架」列表，内容与 output_snapshot.chapters 一致；失败时展示错误信息与「重试」按钮。

### 验证 4：失败时 error_message 有说明、可重试

1. 可选：对未完成参数提取的任务调用 POST framework/run，或注释 .env 中的 DEEPSEEK_API_KEY 后重启 Celery 再点击「开始生成框架」。
2. framework 步骤 **status** 为 **"failed"**，**error_message** 中有明确说明（如「请先完成参数提取」或未配置 API Key 等）。
3. 前端展示「框架生成失败」及错误描述，点击「重试」可再次调用 framework/run。

## 六、常见问题

| 现象 | 可能原因 | 处理办法 |
|------|----------|----------|
| 点击「开始生成框架」返回 202 但 framework 一直 running 或无变化 | 未开 Celery Worker，或 Worker 未加载 tasks.framework | 开一个终端在 backend 下执行 `celery -A celery_app worker -l info -P solo`；检查 `celery_app\__init__.py` 的 `include` 含 `"tasks.framework"`，修改后重启 Worker |
| framework 步骤 status 为 failed，error_message 提示未配置 API Key 或 401 | .env 未配置 DEEPSEEK_API_KEY 或 Key 无效 | 在 `backend\.env` 中设置 `DEEPSEEK_API_KEY=sk-...`（有效密钥）并重启 Celery Worker |
| 参数提取完成后没有「开始生成框架」按钮 | 前端逻辑要求 params 完成且（无 framework 或 framework 为 pending/failed）才显示 | 确认 `frontend\src\pages\TaskDetailPage.tsx` 中 showFrameworkTrigger 条件正确；若 framework 已为 completed 则不会重复显示按钮，可新建任务再测 |
| POST framework/run 返回 400「请先完成参数提取」 | 该任务尚未完成参数提取或 params 无有效 output | 先对该任务完成上传、解析、分析、参数提取（POST extract/run、analyze/run、params/run 并轮询至 params 为 completed），再调 framework/run |
| 框架生成完成但「章节框架」为空或解析不到章节 | LLM 返回的格式未被 parse_framework_text 识别（需「第X章 标题」格式） | 查看 framework 步骤的 output_snapshot 中 chapters 是否为空；若为 [] 多为 LLM 输出格式不符合正则，可查看 Celery 日志或后续优化 prompt |
| 想确认知识库检索是否正常 | 需单独验证 ThinkDoc | 在 backend 下执行 `python test_kb.py`，若 Chunks returned > 0 且有预览即表示检索正常；参见 [ThinkDoc HTTP API](https://doc.bluedigit.ai/docs/developer-guide/http-api) |
| 修改 prompts 或 framework 代码后结果没变 | Worker 未重启 | Celery Worker 需 Ctrl+C 停掉后重新执行 `celery -A celery_app worker -l info -P solo` 才会加载新代码 |
