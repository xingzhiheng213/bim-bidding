# 阶段 3.4 前端审核界面 — 操作与验证指南（新手向）

本文档面向新手，按「需要准备什么 → 一步步操作 → 如何验证」说明阶段 3.4 的完整流程。阶段 3.4 为**前端审核界面**的汇总验收：在框架步骤为 **waiting_user** 时展示章节列表与三个操作按钮（接受并继续、重新生成框架、添加要点）；「添加要点」弹窗提交时调用 save-points 并给出对应提示；「接受并继续」调用 accept，步骤变为 completed 后页面显示「框架已确认，可进入按章生成。」及可选的「已添加 N 个要点」可折叠列表。后端接口与 API 已在阶段 3.2、3.3 完成，本阶段聚焦界面与操作是否齐全、文案是否符合开发计划。

---

## 一、阶段 3.4 要达成什么

- **waiting_user 时界面**：在「步骤 4：框架生成」区块内，展示**章节列表**（可折叠）及副文案「请选择：接受并继续、重新生成或添加要点。」；三个按钮顺序为：**「接受并继续」**、**「重新生成框架」**、**「添加要点」**。
- **「添加要点」**：点击后弹出 Modal，输入多条要点（每行一条），确认按钮为「保存要点」；提交时调用 **save-points**（不调用 accept）；成功后提示「已保存要点，可点击『重新生成框架』使框架按您的要点更新」，弹窗关闭，步骤状态仍为 waiting_user。
- **「接受并继续」**：点击后调用 **accept**，步骤变为 **completed**；页面主文案变为「框架已确认，可进入按章生成。」；若存在 extra_points，展示「已添加 N 个要点」及可折叠的「查看要点」列表。
- **「重新生成框架」**：行为与验收见 [阶段3.2-重生成框架-操作与验证.md](阶段3.2-重生成框架-操作与验证.md)；添加要点后使用见 [阶段3.3-添加要点后继续-操作与验证.md](阶段3.3-添加要点后继续-操作与验证.md)。

**验收标准**（来自开发计划）：框架步骤为 waiting_user 时展示章节列表 + 三操作；添加要点调用 save-points 且提示正确；接受并继续调用 accept，completed 时显示「框架已确认，可进入按章生成。」及可选的「已添加 N 个要点」可折叠列表；整体流程可与 3.2/3.3 联合验收（重生成、加要点、接受后可进入下一阶段）。

## 四、操作步骤（按顺序做）

### 步骤 1～4：与阶段 3.2 相同

确认 Redis、PostgreSQL 已运行；启动后端（终端 1）、Celery Worker（终端 2）、前端（终端 3）。

---

### 步骤 5：进入 waiting_user 并确认 3.4 界面

1. 浏览器打开 <http://localhost:5173>，进入一个**框架已生成、处于等待审核**的任务详情页（或按 3.1 流程跑全流程到「框架已生成，等待您审核。」）。
2. 在「步骤 4：框架生成」区块确认：
   - 主文案为**「框架已生成，等待您审核。」**；
   - 副文案为**「请选择：接受并继续、重新生成或添加要点。」**；
   - 存在**可折叠的「章节框架」**，展开后为章节列表（第 N 章 标题）；
   - 三个按钮可见且顺序为：**接受并继续**、**重新生成框架**、**添加要点**。

---

### 步骤 6：验证「添加要点」弹窗与 save-points 提示

1. 点击**「添加要点」**，应弹出 Modal，标题为「添加要点 / 修改建议」，说明文案含「每行一个要点或修改建议，保存后可点击『重新生成框架』…」，确认按钮为「保存要点」。
2. 在文本框中输入若干行要点（如「增加项目重难点章节」），点击**「保存要点」**。
3. 应出现成功提示**「已保存要点，可点击『重新生成框架』使框架按您的要点更新」**，弹窗关闭；页面仍显示「框架已生成，等待您审核。」，步骤仍为 waiting_user（可刷新或看步骤列表确认）。

---

### 步骤 7：验证「重新生成框架」与「接受并继续」

1. （可选）点击**「重新生成框架」**，等待框架再次生成完成，确认新框架按要点更新且其余章节保持不变（详见阶段 3.3 验收）。
2. 点击**「接受并继续」**，应出现「已接受并继续」类提示；「步骤 4：框架生成」区块主文案变为**「框架已确认，可进入按章生成。」**；三个操作按钮不再显示。

---

### 步骤 8：验证 completed 时「已添加 N 个要点」展示

1. 在上一基础上，若该任务曾保存过要点或 accept 时带有 extra_points，应看到**「已添加 N 个要点：」**及可折叠的**「查看要点」**，展开后为要点列表。
2. 步骤列表中 framework 的**状态**应为 **completed**。

---

## 五、验证（阶段 3.4 验收）

按下面四项逐项检查，都通过即表示阶段 3.4 完成。

### 验证 1：waiting_user 时展示章节列表 + 三按钮且顺序正确

1. 在 framework 为 **waiting_user** 时，「步骤 4：框架生成」区块应显示主文案「框架已生成，等待您审核。」、副文案「请选择：接受并继续、重新生成或添加要点。」、可折叠「章节框架」、三个按钮。
2. 三个按钮顺序为：**接受并继续** → **重新生成框架** → **添加要点**。

---

### 验证 2：添加要点弹窗调用 save-points，提示正确

1. 点击「添加要点」后弹窗内容与「保存要点」按钮符合上文描述。
2. 提交后调用 save-points（可从网络面板或后端日志确认），成功提示为「已保存要点，可点击『重新生成框架』使框架按您的要点更新」，步骤状态仍为 waiting_user。

---

### 验证 3：接受并继续调用 accept，completed 时主文案正确

1. 点击「接受并继续」后调用 accept，framework 步骤变为 **completed**。
2. 页面主文案为「框架已确认，可进入按章生成。」，三按钮不再显示。

---

### 验证 4：completed 且有 extra_points 时展示「已添加 N 个要点」及可折叠列表

1. 当 framework 为 completed 且 output_snapshot 中存在 extra_points 时，应显示「已添加 N 个要点：」及可折叠「查看要点」，展开后为要点列表。
2. 若没有 extra_points，可不显示该区块，仅显示「框架已确认，可进入按章生成。」即可。

## 六、常见问题

| 现象 | 可能原因 | 处理办法 |
|------|----------|----------|
| 看不到三个按钮 | framework 不是 waiting_user（如仍是 running 或已是 completed） | 等框架生成完成或进入一个处于 waiting_user 的任务；确认步骤列表中 framework 状态为 waiting_user |
| 点击「添加要点」后提交变成「接受并继续」 | 前端误将 Modal 确认绑成 accept | 确认「保存要点」回调调用的是 saveFrameworkPoints（save-points），不是 acceptFrameworkStep |
| 保存要点后提示不是「已保存要点，可点击…」 | onSuccess 文案未与计划一致 | 检查 TaskDetailPage 中 savePointsMutation.onSuccess 的 message 文案 |
| completed 时没有「已添加 N 个要点」 | output_snapshot 中无 extra_points 或前端未解析 | 确认曾保存过要点或 accept 时传入了 added_points；确认前端从 output_snapshot 解析 extra_points 并在 completed 时渲染 |
