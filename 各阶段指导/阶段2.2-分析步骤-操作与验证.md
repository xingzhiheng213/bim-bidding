# 阶段 2.2 分析步骤 — 操作与验证指南（新手向）

本文档面向新手，按「需要准备什么 → 一步步操作 → 如何验证」说明阶段 2.2 的完整流程。阶段 2.2 实现**分析步骤**：解析完成后，前端可点击「开始分析」；后端入队 Celery 任务，读取 extract 步骤的文本，调用 LLM（默认 DeepSeek）按 7 个维度深度分析招标文件，将结果写入 **analyze** 步骤的 output_snapshot；前端轮询并展示分析结果或失败信息。

---

## 一、阶段 2.2 要达成什么

- 后端提供 **POST /api/tasks/{task_id}/steps/analyze/run**，在 extract 步骤已完成且含解析文本的前提下入队分析任务，返回 202。
- **Celery Worker** 执行分析任务：读取 extract 步骤的 output_snapshot["text"]，用 `app.prompts` 中的 system/user 模板调用 LLM（默认 deepseek / deepseek-chat，temperature 0.2），将返回内容写入 analyze 步骤的 output_snapshot 为 `{"text": "..."}`；失败时 status 为 failed，error_message 有说明。
- **前端**：解析完成后出现「开始分析」按钮；点击后调用上述接口，分析中显示「分析中，请稍候…」并每约 2 秒轮询；分析完成后展示可折叠的「分析结果」；失败时展示错误信息与「重试」按钮。
- 验收时能在**网页上**或**接口**完成：创建任务 → 上传 → 解析完成 → 点击「开始分析」（或 POST analyze/run）→ 轮询至 analyze 完成 → 看到分析结果长文本；未配置 API Key 时分析失败且有明确提示。

**验收标准**（来自开发计划）：上传并解析后，能点击「开始分析」；分析任务入队后前端轮询直至完成或失败；分析完成后能看到 analyze 步骤的分析结果；失败时能看到错误信息并可重试。

## 三、环境准备

### 1. 确认 Celery 加载了 analyze 任务

打开 `backend\celery_app\__init__.py`，确认 `include` 中包含 `"tasks.analyze"`，例如：

```python
include=["tasks.demo", "tasks.extract", "tasks.analyze"],
```

若没有 `"tasks.analyze"`，请加上并保存；**修改后必须重启 Celery Worker**，否则点击「开始分析」会 202 但任务一直不执行或报「未注册任务」。

### 2. 配置 DeepSeek API Key

1. 打开或创建 `backend\.env`（可从 `.env.example` 复制）。
2. 添加或取消注释一行：
   ```env
   DEEPSEEK_API_KEY=sk-你的DeepSeek密钥
   ```
3. 保存。**若之前 Celery 已启动，修改 .env 后需重启 Celery Worker** 才能读到新环境变量。

可选：若要改用其他模型，可在 .env 中设置：
- `ANALYZE_LLM_PROVIDER=deepseek`（默认）
- `ANALYZE_LLM_MODEL=deepseek-chat`（默认）

### 3. 服务顺序

自 2.2 起与 1.2、1.3 相同，需要 **3 个终端**：后端（uvicorn）、Celery Worker、前端（npm run dev）。详见 [开发前-服务启动顺序.md](开发前-服务启动顺序.md)。

## 四、操作步骤（按顺序做）

### 步骤 1：确认 Redis、PostgreSQL 已运行

- **Redis**：CMD 或 PowerShell 执行 `redis-cli ping`，看到 `PONG`。
- **PostgreSQL**：pgAdmin 能连上本机或服务里 PostgreSQL 为「正在运行」。

若未运行，按 [开发前-服务启动顺序.md](开发前-服务启动顺序.md) 的「二」先启动。

---

### 步骤 2：启动后端（终端 1）

在终端（PowerShell 或 CMD）中执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
uvicorn app.main:app --reload --host 0.0.0.0 --port 8001
```

看到 `Uvicorn running on http://0.0.0.0:8001` 和 `Application startup complete.` 即表示后端就绪。**此窗口不要关**。

---

### 步骤 3：启动 Celery Worker（终端 2）

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
celery -A celery_app worker -l info -P solo
```

看到 `celery@xxx ready.` 即表示 Worker 就绪。**此窗口不要关**。  
若未做这一步，或 `include` 中无 `tasks.analyze`，「开始分析」会返回 202 但 analyze 步骤会一直 running 或报未注册任务。

---

### 步骤 4：启动前端（终端 3）

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\frontend
npm run dev
```

看到 `Local: http://localhost:5173/` 即表示前端就绪。**此窗口不要关**。

---

### 步骤 5：在网页上完成「上传 → 解析 → 分析」

1. 浏览器打开 <http://localhost:5173>，点击**「创建任务」**，进入任务详情页。
2. **上传**：在「上传招标文件」区块选择本机一个 **PDF 或 DOCX**，点击**「上传」**，等待上传成功（会自动触发解析）。
3. **解析**：等待「解析完成，共 xxx 字」和预览出现（若一直「解析中」，检查 Celery 终端是否有报错）。
4. **分析**：
   - 解析完成后应出现**「开始分析」**按钮，点击它。
   - 应出现「分析已入队」或类似提示，随后显示**「分析中，请稍候…」**（页面会约每 2 秒自动轮询）。
5. **等分析完成**（通常几十秒到一两分钟，视文档长度和网络而定）：
   - 应出现**「分析完成。」**和可折叠的**「分析结果」**区块，展开后可看到按 7 个维度（项目基本信息、BIM 技术要求、技术标格式、评分细则、资质与业绩、废标条款与风险点、时间节点）整理的长文本。
6. 页面底部**步骤列表**中应有一条 **step_key 为 "analyze"**、**status 为 "completed"**。

若分析失败（如未配置 DEEPSEEK_API_KEY），会显示红色**「分析失败」**及错误信息，可点击**「重试」**再次触发（仍会失败直到配置好 Key）。

---

### 步骤 6：（可选）用接口验收

若习惯用 curl 或 Postman，可按下面做（把 `任务id` 换成实际数字）：

1. 创建任务：`POST http://localhost:8001/api/tasks`，body `{}`，记下返回的 `id`。
2. 上传文件：`POST http://localhost:8001/api/tasks/{任务id}/upload`，form-data 字段 `file` 为 PDF/DOCX。
3. 触发解析：`POST http://localhost:8001/api/tasks/{任务id}/steps/extract/run`，返回 202。
4. 轮询 `GET http://localhost:8001/api/tasks/{任务id}`，直到 steps 中 extract 的 status 为 completed。
5. 触发分析：`POST http://localhost:8001/api/tasks/{任务id}/steps/analyze/run`，应返回 **202**，body 含 `"message": "分析已入队"`、`"step_key": "analyze"`。
6. 继续轮询 `GET /api/tasks/{任务id}`，直到 steps 中 analyze 的 status 为 completed；检查该步骤的 output_snapshot 为 JSON 字符串，解析后含 `"text"` 键，值为大段分析文本。

## 五、验证（阶段 2.2 验收）

按下面四项逐项检查，都通过即表示阶段 2.2 完成。

### 验证 1：开始分析接口返回 202 且任务入队

1. 在 extract 步骤已完成、且 output_snapshot 含有效 "text" 的前提下，调用 `POST /api/tasks/{task_id}/steps/analyze/run`。
2. 应返回 **202**，body 含 `"message": "分析已入队"`、`"step_key": "analyze"`。
3. 若 extract 未完成或任务不存在，应返回 **400**「请先完成文档解析」或 **404**；Celery 终端应收到 `run_analyze` 任务（收到后若条件不满足，任务会把 analyze 置为 failed）。

### 验证 2：analyze 步骤完成且含分析文本

1. 调用 `GET /api/tasks/{id}`（id 为已调用过 analyze/run 的任务 id）。
2. 在 **steps** 中找到 **step_key 为 "analyze"** 的项。
3. 其 **status** 为 **"completed"**，**output_snapshot** 为合法 JSON，包含 **"text"** 键，值为 LLM 按 7 个维度输出的分析长文本。

### 验证 3：前端能触发、轮询并展示结果

1. 在任务详情页，解析完成后出现「开始分析」按钮；点击后提示入队，并显示「分析中，请稍候…」。
2. 轮询若干次后（不刷新页面）自动变为「分析完成」或「分析失败」。
3. 分析完成时展示「分析完成。」和可折叠的「分析结果」，内容与 output_snapshot["text"] 一致；分析失败时展示错误信息与「重试」按钮。

### 验证 4：失败时 error_message 有说明、可重试

1. 可选：注释掉 .env 中的 DEEPSEEK_API_KEY，重启 Celery，再对已解析任务点击「开始分析」。
2. analyze 步骤 **status** 为 **"failed"**，**error_message** 中有明确说明（如未配置 API Key 或 401）。
3. 前端展示「分析失败」及错误描述，点击「重试」可再次调用 analyze/run（仍会失败直到配置好 Key）。

## 六、常见问题

| 现象 | 可能原因 | 处理办法 |
|------|----------|----------|
| 点击「开始分析」返回 202 但 analyze 一直 running 或无变化 | 未开 Celery Worker，或 Worker 未加载 tasks.analyze | 开一个终端在 backend 下执行 `celery -A celery_app worker -l info -P solo`；检查 `celery_app\__init__.py` 的 `include` 含 `"tasks.analyze"`，修改后重启 Worker |
| analyze 步骤 status 为 failed，error_message 提示未配置 API Key 或 401 | .env 中未配置 DEEPSEEK_API_KEY 或 Key 无效 | 在 `backend\.env` 中设置 `DEEPSEEK_API_KEY=sk-...`（有效密钥），保存后**重启 Celery Worker** |
| 前端没有「开始分析」按钮 | 解析未完成，或 analyze 已是 completed | 等待「解析完成，共 xxx 字」出现后再看；若 analyze 已成功完成，不会重复显示按钮（可新建任务再测） |
| POST analyze/run 返回 400「请先完成文档解析」 | 该任务尚未完成解析或 extract 无 text | 先对该任务完成上传并调用 POST .../steps/extract/run，轮询直到 extract 为 completed 且 output_snapshot 含 "text"，再调 analyze/run |
| 修改 prompts 或 analyze 代码后结果没变 | Worker 未重启 | Celery Worker 需 Ctrl+C 停掉后重新执行 `celery -A celery_app worker -l info -P solo` 才会加载新代码 |
| Celery 启动时报错或收到「unregistered task」 | include 缺少 tasks.analyze 或 tasks.analyze 有语法错误 | 确认 `celery_app\__init__.py` 中 `include` 含 `"tasks.analyze"`；检查 `tasks\analyze.py` 无语法错误，必要时 `python -c "from tasks.analyze import run_analyze"` 验证 |
