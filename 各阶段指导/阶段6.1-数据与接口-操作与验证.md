# 阶段 6.1 数据与接口 — 操作与验证指南（新手向）

本文档面向新手，按「需要准备什么 → 一步步操作 → 如何验证」说明阶段 6.1 的完整流程。阶段 6.1 为**审核专家模块的数据与接口**：支持「要点变更前 / 要点变更后」两段文本对比，提供 `POST /api/compare` 返回结构化 diff；可选地保存重生成前快照，并提供 `GET .../steps/framework/diff`、`GET .../steps/chapters/diff` 从后端取最近一次要点变更的对比数据。

---

## 一、阶段 6.1 要达成什么

- **两段文本对比**：至少支持传入两段文本（original、modified），返回结构化 diff（`{type: 'add'|'del', text: '...'}` 的列表），供后续 6.2/6.3 标红标绿展示。
- **POST /api/compare**：请求体 `{ "original": string, "modified": string }`，响应 `{ "diff": [ {"type": "add"|"del", "text": "..."}, ... ] }`；后端使用行级 diff（difflib），无新依赖。
- **可选：保存「要点变更前」快照**：框架或单章「重新生成」前，将当前内容写入 `task_steps.output_snapshot_before_regenerate`，便于刷新页面后仍能从后端拿到对比数据。
- **可选：GET diff 接口**：`GET /api/tasks/{task_id}/steps/framework/diff` 返回最近一次框架重生成的 original、modified、diff；`GET /api/tasks/{task_id}/steps/chapters/diff?chapter_number=N` 返回第 N 章最近一次重生成的 original、modified、diff。
- **验收标准**（来自开发计划）：能传入两段文本得到结构化 diff；若实现可选部分，则重生成框架或某章后，刷新页面再调用 GET .../diff 能拿到最近一次要点变更的对比数据。

## 三、环境准备

- **仅测 POST /api/compare**：只需后端（uvicorn）运行，无需 Celery、前端。
- **测 GET framework/diff、chapters/diff**：后端运行即可；数据来自已执行过「重新生成框架」或「重新生成本章」的任务（Celery 在重生成时已写入快照）。

---

## 四、操作步骤（按顺序做）

### 步骤 1：启动后端

1. 打开终端，进入 `backend` 目录，激活虚拟环境，执行：`uvicorn app.main:app --reload --host 0.0.0.0 --port 8000`。
2. 浏览器访问 http://localhost:8000/health，应看到 `{"status":"ok"}`。

---

### 步骤 2：验证 POST /api/compare（必做）

**目的**：确认传入两段文本能返回结构化 diff。

**方式 A：curl（推荐）**

在 CMD 或 PowerShell 中执行（一行）：

```bash
curl -X POST "http://localhost:8000/api/compare" -H "Content-Type: application/json" -d "{\"original\": \"hello\nworld\", \"modified\": \"hello\nnew\nworld\"}"
```

**预期结果**：返回 JSON 中含 `"diff"` 数组，且至少有一项为 `{"type":"add","text":"new\n"}`（表示新增了一行）。

**方式 B：浏览器 + 开发者工具**

1. 打开浏览器开发者工具（F12），切到「Network」。
2. 在 Console 中执行（或使用可发 POST 的扩展）：
   ```javascript
   fetch('http://localhost:8000/api/compare', {
     method: 'POST',
     headers: { 'Content-Type': 'application/json' },
     body: JSON.stringify({ original: 'hello\nworld', modified: 'hello\nnew\nworld' })
   }).then(r => r.json()).then(console.log);
   ```
3. 查看返回对象中的 `diff` 数组，应包含 `type` 为 `add` 或 `del` 的项。

---

### 步骤 3：验证 GET /api/tasks/{task_id}/steps/framework/diff（可选）

**前提**：至少有一个任务已经做过一次「重新生成框架」（在前端对该任务点击「重新生成框架」并等 Celery 跑完）。

1. 记下该任务的 **任务 ID**（如 5、10，可从任务详情页 URL 或列表看到）。
2. 在浏览器地址栏输入（将 `5` 换成你的任务 ID）：
   ```
   http://localhost:8000/api/tasks/5/steps/framework/diff
   ```
3. 回车后，页面应显示 JSON，包含 `original`、`modified`、`diff` 三个字段；`original` 为重生成前的框架文本（每行一章），`modified` 为重生成后的框架文本，`diff` 为增删片段列表。

若返回 404 且提示「无框架重生成前后数据」，说明该任务尚未做过「重新生成框架」，需先在前端对该任务执行一次「重新生成框架」后再测。

---

### 步骤 4：验证 GET /api/tasks/{task_id}/steps/chapters/diff?chapter_number=N（可选）

**前提**：至少有一个任务已完成「按章生成」，且对其中某一章（如第 1 章）做过一次「重新生成本章」。

1. 记下该任务的 **任务 ID** 和已重生成的 **章节号**（如 1、3）。
2. 在浏览器地址栏输入（将 `5` 换成任务 ID，`1` 换成章节号）：
   ```
   http://localhost:8000/api/tasks/5/steps/chapters/diff?chapter_number=1
   ```
3. 回车后，页面应显示 JSON，包含 `original`、`modified`、`diff`；`original` 为该章重生成前正文，`modified` 为该章重生成后正文，`diff` 为增删片段列表。

若返回 404 且提示「无第 N 章重生成前后数据」，需先对该任务该章执行一次「重新生成本章」后再测。

## 五、验证（阶段 6.1 验收）

按下面两项逐项检查，必做通过即满足 6.1 最低验收；可选通过则说明「刷新后仍可对比」已可用。

### 验证 1：POST /api/compare 返回结构化 diff（必做）

1. 使用步骤 2 的 curl 或浏览器方式，传入 `original` 与 `modified` 两段文本。
2. 响应中 `diff` 为数组，每项含 `type`（`"add"` 或 `"del"`）和 `text`。
3. 能区分新增与删除（例如仅新增一行时只有 `type: "add"`；既有改行则同时出现 `del` 与 `add`）。

### 验证 2：GET framework/diff 与 chapters/diff 能拿到最近一次对比数据（可选）

1. 对某任务执行一次「重新生成框架」后，访问 `GET /api/tasks/{任务ID}/steps/framework/diff`，返回 200 且含 `original`、`modified`、`diff`。
2. 对某任务某章执行一次「重新生成本章」后，访问 `GET /api/tasks/{任务ID}/steps/chapters/diff?chapter_number=N`，返回 200 且含该章的 `original`、`modified`、`diff`。

## 六、常见问题

| 现象 | 可能原因 | 处理办法 |
|------|----------|----------|
| POST /api/compare 返回 404 | 路径错误或后端未启动 | 确认 URL 为 `http://localhost:8000/api/compare`（无 `tasks`）；确认后端已启动 |
| POST 返回 422 | 请求体不是合法 JSON 或缺少字段 | 确保 Body 为 JSON，且包含 `original`、`modified` 两个字符串字段（可为空字符串） |
| GET framework/diff 返回 404「无框架重生成前后数据」 | 该任务从未做过「重新生成框架」 | 在前端对该任务点一次「重新生成框架」，等完成后再次访问 GET 地址 |
| GET chapters/diff 返回 404「无第 N 章重生成前后数据」 | 该章从未做过「重新生成本章」 | 对该任务该章点一次「重新生成本章」，等完成后再次访问 GET 地址 |
| 启动后端报错与 task_steps 表相关 | 旧库缺少 `output_snapshot_before_regenerate` 列 | 应用启动时会自动执行 `ALTER TABLE task_steps ADD COLUMN ...`；若仍报错，可手动在 PostgreSQL 执行：`ALTER TABLE task_steps ADD COLUMN output_snapshot_before_regenerate TEXT;` |

## 七、小结：阶段 6.1 检查清单

- [ ] **验证 1**：POST /api/compare 传入两段文本，返回的 `diff` 为列表，每项含 `type` 与 `text`，能区分 add/del。
- [ ] **验证 2（可选）**：GET .../steps/framework/diff 在「重新生成框架」后能返回 original、modified、diff。
- [ ] **验证 2（可选）**：GET .../steps/chapters/diff?chapter_number=N 在「重新生成本章」后能返回该章的 original、modified、diff。
