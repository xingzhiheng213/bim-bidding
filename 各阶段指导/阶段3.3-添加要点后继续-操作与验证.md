# 阶段 3.3 添加要点后继续 — 操作与验证指南（新手向）

本文档面向新手，按「需要准备什么 → 一步步操作 → 如何验证」说明阶段 3.3 的完整流程。阶段 3.3 实现**添加要点后继续**：**思路**为「添加要点」是针对**当前生成框架的修改建议」**；保存要点后，用户点击「重新生成框架」时，LLM 只收到「当前框架 + 用户要点」，按要点增删改框架（未提及的章节必须原样保留）；用户点击「接受并继续」后，框架步骤置为 completed，可进入按章生成。

---

## 一、阶段 3.3 要达成什么

### 思路转变（与初版计划的区别）

- **初版设想**：「添加要点」后要点带入**按章生成**环节。
- **当前实现**：「添加要点」是对**当前框架的修改建议**。用户保存要点后，应点击「**重新生成框架**」，由 LLM 根据「当前框架 + 用户要点」生成新框架（例如用户写「增加项目重难点章节，其他章节保持不变」，则只新增该章，其余标题一字不改）；满意后再点「接受并继续」进入下一阶段。

### 后端

- **POST /api/tasks/{task_id}/steps/framework/save-points**  
  - Body：`{ "added_points": ["要点1", "要点2"] }`。  
  - 仅当 framework 为 **waiting_user** 时允许：将 `added_points` 写入 framework 步骤的 `output_snapshot.extra_points`，**不改变步骤状态**。用于「先保存修改建议，再点重新生成框架」。
- **POST /api/tasks/{task_id}/steps/framework/accept**  
  - Body 可选：`{ "added_points": ["要点1", "要点2"] }`。  
  - 仅当 framework 为 **waiting_user** 时允许：若提供 `added_points` 则合并到 `extra_points`，然后将步骤状态置为 **completed**，表示用户已确认框架，可进入按章生成。
- **框架生成任务（run_framework）**  
  - 若当前 framework 的 `output_snapshot` 中存在 **extra_points**：构建 prompt 时**只**给 LLM「当前框架 + 用户要点」，不给招标分析、BIM 要求、知识库等，减少干扰。  
  - 系统提示硬约束：只对用户要求中明确提到的部分做增删改；用户未提及的章节必须与当前框架完全一致，不得改写、合并、删除、调序。  
  - 有用户要点时使用较低 temperature（如 0.1），减少对未要求部分的随意改写。  
  - 若**没有** extra_points：与原先一致，用「招标分析 + BIM 要求 + 知识库」做完整生成。

### 前端

- **「添加要点」**：弹窗输入多条要点，提交时调用 **save-points**；成功后提示「已保存要点，可点击『重新生成框架』使框架按您的要点更新」（不调用 accept，不改变状态）。
- **「重新生成框架」**：沿用阶段 3.2，调用 regenerate；有保存的要点时，Celery 跑 run_framework 会读 extra_points，只发「当前框架+要点」给 LLM。
- **「接受并继续」**：调用 **accept**，框架步骤变为 completed；页面在 completed 时显示「框架已确认，可进入按章生成。」，若有 extra_points 可展示「已添加 N 个要点」及可折叠要点列表。

**验收标准**：在 waiting_user 时可保存要点（save-points）；保存后点「重新生成框架」能按要点更新框架且其他章节保持不变；点「接受并继续」后步骤为 completed，可进入下一阶段；前端文案与操作流程符合上述思路。

## 四、操作步骤（按顺序做）

### 步骤 1～4：与阶段 3.2 相同

确认 Redis、PostgreSQL 已运行；启动后端（终端 1）、Celery Worker（终端 2）、前端（终端 3）。若刚改过 `prompts.py` 或 `framework.py`，请先重启 Celery Worker 再测试。

---

### 步骤 5：添加要点并保存（不改变状态）

1. 浏览器打开 <http://localhost:5173>，进入一个**框架已生成、处于 waiting_user** 的任务详情页。
2. 点击**「添加要点」**，在弹窗中输入若干条修改建议，例如：
   - 「增加关于项目重难点的章节」
   - 「其他章节保持不变」
3. 点击弹窗中的确认（如「保存要点」），应调用 **save-points**，并出现提示「已保存要点，可点击『重新生成框架』使框架按您的要点更新」。
4. **确认状态未变**：页面仍显示「框架已生成，等待您审核。」，framework 步骤仍为 **waiting_user**；GET /api/tasks/{id} 中 framework 的 output_snapshot 应包含 **extra_points** 数组。

---

### 步骤 6：重新生成框架（按要点更新）

1. 在上一基础上，点击**「重新生成框架」**。
2. 等待 Celery 执行完成（轮询至 framework 再次 **waiting_user** 或 failed）。
3. **验收**：新框架应体现用户要点（例如新增了「项目重难点」相关章节），且**未在要点中提及的章节**应与之前保持一致（标题、顺序不变）。若发现其他章节被改写，检查 prompt 是否使用「仅当前框架+要点」及硬约束「未提及章节必须原样」，并确认 Celery 已重启。

---

### 步骤 7：接受并继续

1. 对当前框架满意后，点击**「接受并继续」**，应调用 **accept**。
2. 页面应变为「框架已确认，可进入按章生成。」；若有 extra_points，可显示「已添加 N 个要点」及可折叠要点列表。
3. **后端确认**：GET /api/tasks/{id} 中 framework 步骤 **status** 应为 **completed**；output_snapshot 中 chapters 为当前框架，extra_points 保留（若之前有保存或 accept 时传入）。

---

### 步骤 8：（可选）用接口验证 save-points 与 accept

1. 对 framework 为 **waiting_user** 的任务：  
   **POST** /api/tasks/{id}/steps/framework/save-points，body：`{"added_points":["增加项目重难点章节","其他章节保持不变"]}`。  
   应返回 200；GET 任务详情中 framework.output_snapshot.extra_points 应有两条，且 status 仍为 waiting_user。
2. **POST** /api/tasks/{id}/steps/framework/accept（body 可为 `{}` 或再次带 `added_points`）。  
   应返回 200；framework.status 应为 completed。

---

## 五、验证（阶段 3.3 验收）

### 验证 1：保存要点不改变状态，重新生成后按要点更新且其余不变

1. 在 waiting_user 下添加要点并保存（前端调用 save-points）。
2. 确认 framework 仍为 waiting_user，output_snapshot 中有 extra_points。
3. 点击「重新生成框架」，等待完成。
4. 新框架满足：用户要点得到体现（如新增某章）；未提及的章节与保存要点前的框架一致（标题、顺序不变）。

---

### 验证 2：接受并继续后为 completed，前端展示正确

1. 在 waiting_user 下点击「接受并继续」，调用 accept。
2. framework 步骤 status 为 **completed**。
3. 前端显示「框架已确认，可进入按章生成。」；若有 extra_points，显示「已添加 N 个要点」及可折叠列表。

---

### 验证 3：无用户要点时框架生成行为不变

1. 对从未保存过要点的任务（或新任务跑通到 framework 第一次生成），点击「重新生成框架」。
2. 应仍使用「招标分析 + BIM 要求 + 知识库」完整生成，行为与阶段 2.4/3.2 一致，不因 3.3 改动而改变。

---

## 六、常见问题

| 现象 | 可能原因 | 处理办法 |
|------|----------|----------|
| 添加要点后点「重新生成」但框架没按要点改 | Celery 未重启，仍用旧 prompt | 修改 prompts.py 或 framework.py 后需重启 Celery Worker |
| 其他章节被改写 | 模型未遵守「未提及章节原样」 | 确认 FRAMEWORK_SYSTEM_WITH_USER_POINTS 中硬约束存在，且有要点时 temperature 已降低（如 0.1） |
| 保存要点后状态变成 completed | 前端误调了 accept 而非 save-points | 确认「添加要点」弹窗确认调用的是 saveFrameworkPoints（save-points），不是 accept |
| save-points 返回 400 | framework 不是 waiting_user | 仅在 waiting_user 时允许 save-points |
