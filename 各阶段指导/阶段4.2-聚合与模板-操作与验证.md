# 阶段 4.2 聚合与模板 — 操作与验证指南（新手向）

本文档面向新手，按「需要准备什么 → 一步步操作 → 如何验证」说明阶段 4.2 的完整流程。阶段 4.2 实现**聚合与模板**：参考 Dify 文档聚合模板，将**项目信息（project_info_md）+ 各章内容 + 风险点附录**拼成一份 Markdown；章节顺序与分隔符按业务调整。本阶段仅实现聚合逻辑（`app/assembler.py`），供后续 4.3 DOCX 导出调用，不涉及 API 或 Celery 任务。

---

## 一、阶段 4.2 要达成什么

### 目标

- **`project_info_to_markdown(project_info: dict) -> str`**：将 params.project_info（dict）转为可读 Markdown 列表，键名映射（name→项目名称、scale→项目规模等）。
- **`assemble_full_markdown(task_id: int, db: Session) -> str`**：读取 params、framework、chapters 步骤输出，拼接为完整 Markdown。
- **文档结构**：`# BIM技术标完整文档` → `## 项目信息` → `---` → 各章正文（按框架顺序）→ `## 附录` → `### 风险点提醒`。
- **缺失章节**：若某章在 chapters 步骤中未生成，插入占位「（第 X 章 xxx 内容待生成）」。

### 数据来源

| 数据 | 来源步骤 | 字段 |
|------|----------|------|
| project_info | params | output_snapshot.project_info（dict） |
| risk_points | params | output_snapshot.risk_points（list） |
| 章节顺序 | framework | output_snapshot.chapters（list，含 number/title/full_name） |
| 章节正文 | chapters | output_snapshot.chapters（dict，key 为章节号字符串） |

**验收标准**（来自开发计划）：能根据 params、framework、chapters 步骤输出拼成一份 Markdown；结构为项目信息 + 各章内容 + 风险点附录；章节顺序与 framework 一致；为 4.3 DOCX 导出提供可调用接口。

## 四、操作步骤（按顺序做）

### 步骤 1：确认 PostgreSQL 已运行

- PostgreSQL 需运行，以便 `assemble_full_markdown` 能连接数据库读取任务步骤。
- 可在「服务」中查看 PostgreSQL 服务是否已启动，或通过 pgAdmin 连接本机验证。

---

### 步骤 2：启动后端虚拟环境

打开 CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
```

---

### 步骤 3：测试 project_info_to_markdown（无需任务数据）

此测试不依赖数据库，仅验证项目信息转 Markdown 的逻辑：

```powershell
python test_assembler.py
```

**预期输出**：

```
project_info_to_markdown: OK
To verify assemble_full_markdown, run: python test_assembler.py <task_id>
```

若出现 `AssertionError`，说明 `project_info_to_markdown` 有问题。

---

### 步骤 4：准备一个已完成按章生成的任务（用于 assemble_full_markdown）

需要有一个**已完成 params、framework（接受并继续）、chapters** 的任务，并拿到其 `task_id`。

#### 方法 A：从页面创建并跑完流程

1. 按 [开发前-服务启动顺序.md](开发前-服务启动顺序.md) 启动后端、Celery、前端。
2. 打开 http://localhost:5173，点击「创建任务」，进入任务详情。
3. 依次完成：上传招标文件 → 解析 → 分析 → 参数提取 → 框架生成 → **接受并继续** → 按章生成。
4. 浏览器地址如 `http://localhost:5173/tasks/31`，其中 **31** 即为 `task_id`。

#### 方法 B：从任务列表拿 task_id

1. 打开 http://localhost:5173，在任务列表中找一个**已完成按章生成**的任务。
2. 点击进入，从 URL 中获取 `task_id`。

#### 方法 C：用 API 查询

```powershell
curl http://localhost:8000/api/tasks
```

返回的任务列表中，找一个你确认已完成按章生成的任务，取其 `id` 作为 `task_id`。

---

### 步骤 5：测试 assemble_full_markdown（使用真实 task_id）

在 backend 目录下执行（将 `31` 替换为你的 `task_id`）：

### 步骤 6：（可选）将聚合结果保存到文件检查

想完整查看聚合出的 Markdown 内容时，可执行：

```powershell
python -c "
from app.database import SessionLocal
from app.assembler import assemble_full_markdown

db = SessionLocal()
try:
    md = assemble_full_markdown(31, db)  # 替换为你的 task_id
    with open('assembled_output.md', 'w', encoding='utf-8') as f:
        f.write(md)
    print('已保存到 assembled_output.md，长度:', len(md))
finally:
    db.close()
"
```

然后打开 `backend/assembled_output.md` 查看内容，确认结构为：`# BIM技术标完整文档` → `## 项目信息` → `---` → 各章 → `## 附录` → `### 风险点提醒`。

## 六、常见问题

| 现象 | 可能原因 | 处理办法 |
|------|----------|----------|
| `请先完成参数提取` | 该任务 params 未 completed | 先完成参数提取步骤 |
| `请先完成框架并接受框架` | framework 未 completed 或未点「接受并继续」 | 完成框架生成并点击「接受并继续」 |
| `请先完成按章生成` | chapters 未 completed | 先完成按章生成 |
| `参数/框架/按章生成步骤输出格式异常` | 对应步骤的 output_snapshot 不是合法 JSON | 检查数据库或重新跑该步骤 |
| 测试脚本报 `ModuleNotFoundError` | 未在 backend 目录或未激活 venv | `cd backend`、`.venv\Scripts\activate` 后再运行 |
