# 阶段 1.2 文档解析 — 操作与验证指南（新手向）

本文档面向新手，按「需要准备什么 → 一步步操作 → 如何验证」说明阶段 1.2 的完整流程。阶段 1.2 实现**文档解析**：上传完成后调用接口触发解析，Celery 任务读取已上传文件，用 PyMuPDF / python-docx 提取纯文本，将结果写入 **extract** 步骤的 output_snapshot（不包含前端「上传后自动触发解析」与解析结果展示，前端在 1.3）。

---

## 一、阶段 1.2 要达成什么

- 后端提供 **POST /api/tasks/{task_id}/steps/extract/run**，在 upload 步骤已完成的前提下入队解析任务，返回 202。
- **Celery Worker** 执行解析任务：根据 upload 步骤的 stored_path 找到文件，调用 `parse_document` 提取纯文本（支持 .pdf、.docx；.doc 暂不支持）。
- 创建或更新该任务的 **extract** 步骤：成功时 status 为 completed，output_snapshot 为 `{"text": "..."}`；失败时 status 为 failed，error_message 有说明。
- 验收时能用 **curl 或 Postman** 完成：创建任务 → 上传文件（1.1）→ 调用「开始解析」→ 轮询 GET 任务详情，看到 extract 步骤完成且含解析出的文字。

**验收标准**（来自开发计划）：上传一个 PDF 或 DOCX 后，能通过接口触发「开始解析」，轮询任务详情可见 extract 步骤完成且含解析文本（或字数）；无报错。

## 三、环境准备

- **后端依赖**：阶段 1.2 增加了 `pymupdf`（PDF）、`python-docx`（DOCX）。在 `backend` 目录下、激活虚拟环境后执行：  
  `pip install -r requirements.txt`  
  或至少：`pip install pymupdf python-docx`
- **服务顺序**：自 1.2 起需要 **3 个终端**：后端（uvicorn）、Celery Worker、前端（可选）。详见 [开发前-服务启动顺序.md](开发前-服务启动顺序.md)。
- **若之前只开了后端和前端**：请**新开一个终端**专门跑 Celery，并保持后端、Celery、前端（若用）三个窗口都开着；**不要**在未重启的情况下漏开 Celery，否则 POST extract/run 会 202 但任务一直不执行。

## 四、操作步骤（按顺序做）

### 步骤 1：确认 Redis、PostgreSQL 已运行

- **Redis**：CMD 或 PowerShell 执行 `redis-cli ping`，看到 `PONG`。
- **PostgreSQL**：pgAdmin 能连上本机或服务里 PostgreSQL 为「正在运行」。

若未运行，按 [开发前-服务启动顺序.md](开发前-服务启动顺序.md) 的「二」先启动。

---

### 步骤 2：启动后端（终端 1）

在终端（PowerShell 或 CMD）中执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload --host 0.0.0.0 --port 8001
```

看到 `Uvicorn running on http://0.0.0.0:8001` 和 `Application startup complete.` 即表示后端就绪。**此窗口不要关**。

---

### 步骤 3：启动 Celery Worker（终端 2）— 1.2 必须

**再开一个** CMD 或 PowerShell，执行：

```powershell
cd d:\标书工作流\backend
.venv\Scripts\activate
celery -A celery_app worker -l info -P solo
```

看到 `celery@xxx ready.` 和 `transport: redis://...` 即表示 Worker 就绪。**此窗口不要关**。  
若不做这一步，后面「开始解析」会返回 202，但 extract 步骤会一直处于 running 或从未更新。

---

### 步骤 4：创建一条任务（第三个终端或用 Postman）

再开一个终端（或使用 Postman），执行（创建任务）：

```powershell
curl -X POST http://localhost:8001/api/tasks -H "Content-Type: application/json" -d "{}"
```

或 PowerShell：

```powershell
Invoke-RestMethod -Method Post -Uri "http://localhost:8001/api/tasks" -ContentType "application/json" -Body "{}"
```

记下返回 JSON 里的 **id**（例如 `5`）。后面把「任务 id」都换成这个数字。

---

### 步骤 5：上传文件到该任务

将下面命令里的 **任务 id** 和 **文件路径** 换成你的实际值后执行。

```powershell
curl -X POST http://localhost:8001/api/tasks/5/upload -F "file=@C:\Users\你的用户名\Desktop\你的文件.pdf"
```

- 把 **`/5/`** 换成步骤 4 得到的 id。
- 把 **`@C:\...\你的文件.pdf`** 换成你电脑上 PDF 或 DOCX 的**完整路径**。

成功时返回 **201**，body 中应有 `"step_key": "upload"`、`"status": "completed"` 等。

---

### 步骤 6：调用「开始解析」

执行（把 `5` 换成你的任务 id）：

```powershell
curl -X POST http://localhost:8001/api/tasks/5/steps/extract/run
```

或 PowerShell：

```powershell
Invoke-RestMethod -Method Post -Uri "http://localhost:8001/api/tasks/5/steps/extract/run"
```

成功时返回 **202**，body 中应有 `"message": "解析已入队"`、`"step_key": "extract"`。  
此时可在 **Celery 所在终端**看到类似：`Task tasks.extract.run_extract [xxx] received`，随后 `succeeded`。

---

### 步骤 7：轮询任务详情，确认 extract 完成

多次执行（把 `5` 换成你的任务 id）：

```powershell
curl http://localhost:8001/api/tasks/5
```

或 PowerShell：

```powershell
Invoke-RestMethod -Uri "http://localhost:8001/api/tasks/5"
```

在返回的 JSON 里找到 **steps** 中 **step_key 为 "extract"** 的那条：

- **status 为 "completed"**：表示解析成功。**output_snapshot** 为一段 JSON 字符串，解析后应含 **"text"** 键，值为整篇提取的纯文本（可看字数或前几百字）。
- **status 为 "failed"**：看 **error_message**，如「暂不支持 .doc」「文件不存在」等，按提示处理。

若一开始 extract 为 **running**，隔几秒再请求一次，直到变为 completed 或 failed。

---

### 步骤 8：（可选）看 Celery 终端

在跑 Celery 的窗口里，步骤 6 之后应出现：

- `Task tasks.extract.run_extract [xxx] received`
- `Task tasks.extract.run_extract [xxx] succeeded in 0.xxs`

有这两行说明解析任务已被 Worker 执行完毕。

## 五、验证（阶段 1.2 验收）

按下面三项逐项检查，都通过即表示阶段 1.2 完成。

### 验证 1：开始解析接口返回 202 且任务入队

1. 在 upload 步骤已完成的前提下，调用 `POST /api/tasks/{task_id}/steps/extract/run`。
2. 应返回 **202**，body 含 `"message": "解析已入队"`、`"step_key": "extract"`。
3. 若 upload 未完成或任务不存在，应返回 **400** 或 **404**；Celery 终端应收到 `run_extract` 任务（收到后若 upload 未完成，任务会把 extract 置为 failed）。

---

### 验证 2：extract 步骤完成且含解析文本

1. 调用 `GET /api/tasks/{id}`（id 为已调用过 extract/run 的任务 id）。
2. 在 **steps** 中找到 **step_key 为 "extract"** 的项。
3. 其 **status** 为 **"completed"**，**output_snapshot** 为合法 JSON，包含 **"text"** 键，值为从 PDF/DOCX 提取的纯文本（可为空字符串，但不缺键）。

---

### 验证 3：失败时 error_message 有说明

1. 可选：用 .doc 文件上传（或故意传错误路径），再调用 extract/run；或对未上传文件的任务调用 extract/run。
2. 轮询 GET 任务详情后，extract 步骤 **status** 为 **"failed"**，**error_message** 中有明确说明（如「请先完成上传」「暂不支持 .doc」等）。

## 六、常见问题

| 现象 | 可能原因 | 处理办法 |
|------|----------|----------|
| POST extract/run 返回 202 但 extract 一直 running 或没有 extract 步骤 | 未开 Celery Worker 或 Worker 未加载 tasks.extract | 开一个终端，在 backend 下执行 `celery -A celery_app worker -l info -P solo`；确认启动日志里有 `tasks.extract` |
| POST extract/run 返回 400「请先完成文件上传」 | 该任务尚未上传文件或 upload 步骤未完成 | 先对该任务调用 POST .../upload 上传 PDF 或 DOCX，再调 extract/run |
| extract 步骤 status 为 failed，error_message 为「暂不支持 .doc」 | 当前仅支持 .pdf 和 .docx | 请上传 .docx 或 PDF 后重新上传再解析 |
| 后端或 Celery 报错 ModuleNotFoundError: pymupdf / docx | 未安装 1.2 依赖 | 在 backend 下执行 `pip install -r requirements.txt` 或 `pip install pymupdf python-docx`，并重启 uvicorn 与 Celery |
| 修改代码后 extract 结果没变 | Worker 未重启 | Celery Worker 需 Ctrl+C 停掉后重新执行 `celery -A celery_app worker -l info -P solo` 才会加载新代码 |
