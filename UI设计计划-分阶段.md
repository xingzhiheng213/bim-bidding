# BIM 标书生成 App — UI 设计分阶段计划

本文档将 UI 设计/改版拆成可独立交付的阶段，每阶段有明确目标、交付物和验收标准，便于按计划逐步推进，从「开发版界面」过渡到「可对外使用的产品界面」。

---

## 设计原则与定位

- **在现有功能上做视觉与体验升级**：不改变现有业务流程与接口，只改布局、样式、信息层次与交互反馈。
- **设计系统优先**：先建立颜色、字体、间距、圆角等 token，再改页面，保证风格统一、后续迭代可维护。
- **任务流程可读性优先**：任务详情页是核心，优先让「当前到哪一步、该做什么」一目了然；其次再打磨首页、设置、对比等。
- **各阶段可独立验收**：每阶段完成后即可上线或演示该部分；设计稿与实现可分批交付（如先出设计再开发，或设计+实现一阶段一结）。

---

## 当前界面可改进点（参考）

| 方面 | 当前情况 | 可改进方向 |
|------|----------|------------|
| 布局与导航 | 每页各自写 Layout/Header，导航重复、无高亮 | 抽公共 AppLayout，统一顶部导航与当前页高亮 |
| 样式与品牌 | 几乎无自定义样式，Ant Design 默认蓝、大量内联 style | 设计系统（色板、字体、间距）+ ConfigProvider theme，减少内联 |
| 任务详情 | 步骤用表格+多块 Collapse，流程感弱；操作按钮分散 | 步骤条/时间线 + 按步骤分卡片，等待审核态突出 |
| 首页与列表 | 标题重复、有「后端状态」等开发信息；列表偏工具感 | 弱化/移除开发信息；优化空态与列表展示（可考虑卡片） |
| 设置/对比 | 功能完整但为默认表单与双栏布局 | 分区卡片、状态提示更清晰；对比页布局与 diff 样式统一到设计系统 |
| 反馈与状态 | 依赖 Ant 默认 Message/Spin | 统一加载态、错误态、空态样式与文案 |

以上在分阶段实现时按需采纳，不必一次做完。

| 层级 | 技术选型 |
|------|----------|

## 阶段 0：设计系统与基础

**目标**：建立颜色、字体、间距、圆角等设计 token，并通过 Ant Design ConfigProvider 应用到全局，为后续页面改版提供统一基础。

### 0.1 设计 token 定义

- [ ] **颜色**：主色（Primary，用于主按钮、链接、当前步骤等）；成功 / 进行中 / 警告 / 错误 的语义色；中性色（背景、边框、文字层级）。可写在设计说明或 `frontend/src/theme/tokens.ts`（或等价文件）。
- [ ] **字体与层级**：标题 H1–H5、正文、辅助说明的字号与字重；行高与段落间距。与 Ant Design Typography 对应关系写清。
- [ ] **间距**：统一间距尺度（如 4 / 8 / 16 / 24 / 32 px），用于 padding、margin、组件间距。
- [ ] **圆角**：按钮、卡片、输入框等组件的 border-radius 约定。

**交付物**：  
- 设计 token 文档或代码（色板、字体、间距、圆角），团队可据此实现。

### 0.2 Ant Design 主题接入

- [ ] 使用 `ConfigProvider` 的 `theme` 配置：将 Primary 色、圆角、组件尺寸等映射到 Ant 的 token（如 `colorPrimary`、`borderRadius` 等）。
- [ ] 在应用根节点（如 `main.tsx` 或 `App.tsx`）包一层 `ConfigProvider`，确保全局生效。
- [ ] 减少页面级内联 `style`，优先用 token 或 CSS 变量；新写样式尽量不写死色值与间距。

**交付物**：  
- 主题配置文件 + 根节点 ConfigProvider；本地运行可见主色、圆角等已应用。

### 0.3 步骤状态与按钮约定

- [ ] **步骤状态**：为 pending / running / waiting_user / completed / failed 约定展示形式（如标签色、图标、简短文案），并在设计系统文档中固定。
- [ ] **主按钮 / 次按钮**：主要操作（创建任务、开始生成、接受并继续等）用主按钮；次要操作（取消、添加要点、重试等）用次按钮或链接，风格与 token 一致。

**验收标准**：  
- 设计 token 已文档化或代码化；Ant 主题已接入且主色/圆角等在页面中生效；步骤状态与按钮有明确约定，便于阶段 2 任务详情页使用。

## 阶段 1：布局与导航

**目标**：抽公共布局与导航，各页面只负责内容区；导航高亮当前页；移除或弱化开发向信息。

### 1.1 公共布局组件

- [ ] 新增 `AppLayout`（或等价组件）：包含顶部 Header + 主内容区 Content；子页面通过 `Outlet` 或 `children` 渲染在 Content 内。
- [ ] 在路由层统一使用 `AppLayout`：仅首页、设置、对比、任务详情等需要布局的页面作为 Layout 的子路由，保证全站布局一致。

**交付物**：  
- `AppLayout` 组件 + 路由结构调整；所有现有页面在统一布局下正常显示。

### 1.2 顶部导航

- [ ] 导航项：**首页**（任务列表）、**设置**、**对比**；可根据需要增加「创建任务」入口（或保留首页内按钮）。
- [ ] 当前页高亮：根据当前路由高亮对应导航项（样式与设计系统一致）。
- [ ] 品牌/标题：在 Header 左侧保留「BIM 标书生成」或产品名，样式使用设计 token。

**交付物**：  
- 顶部导航实现完成；切换路由时高亮正确。

### 1.3 开发向信息处理

- [ ] 首页：移除或隐藏「后端状态：ok」等开发用文案；若需保留，可收到设置页「调试」区块或仅开发环境显示。
- [ ] 其他页面：检查并移除仅面向开发的提示或冗余标题（如内容区重复大标题），保持信息简洁。

**验收标准**：  
- 全站使用同一套布局与导航；当前页高亮正确；开发向信息已移除或可配置隐藏。

## 阶段 2：任务详情页 — 流程可视化

**目标**：任务详情页用「步骤条/时间线 + 分步卡片」表达流程，使用户能快速理解「当前到哪一步、该做什么」；框架等待审核时视觉突出。

### 2.1 步骤条或时间线

- [ ] 在任务详情页顶部或侧边增加**步骤条**（Ant Design Steps）或**时间线**：上传 → 解析 → 分析 → 参数 → 框架 → 按章生成 → 导出。
- [ ] 步骤条状态与后端一致：pending / running / waiting_user / completed / failed 映射到 Steps 的对应状态（wait / process / finish / error）；`waiting_user` 可用「待确认」类文案与样式区分。
- [ ] 点击步骤可滚动到对应区块（可选），便于长页导航。

**交付物**：  
- 任务详情页具备步骤条/时间线；状态与后端同步；样式符合设计系统。

### 2.2 按步骤分块展示

- [ ] 每个步骤对应一个**卡片或区块**：内含步骤名称、状态标签、简短说明、该步主要操作按钮（如上传、开始解析、重新生成框架、开始按章生成、下载 Word）。
- [ ] 详情内容（解析结果、分析文本、框架列表、章节预览等）放在对应卡片内，可折叠或默认收起，避免一屏信息过载。
- [ ] 按钮仅在该步骤可操作时展示（如解析完成才显示「分析」）；禁用或不可用状态有明确说明。

**交付物**：  
- 任务详情页按步骤分块；操作入口清晰；长内容可折叠。

### 2.3 框架「等待审核」态突出

- [ ] 当框架步骤为 `waiting_user` 时：该步骤在步骤条中突出（如不同边框色或背景色）；对应卡片内集中展示「接受并继续」「重新生成框架」「添加要点」。
- [ ] 可选：顶部或卡片内增加简短提示文案（如「请确认框架或添加要点后继续」），与设计系统语义色一致。

**验收标准**：  
- 打开任务详情页可一眼看到步骤进度与当前步骤；每步操作入口明确；框架待审核时视觉与操作集中、不分散。

## 阶段 3：首页与任务列表

**目标**：首页信息层次清晰，任务列表易读；空态友好；可选卡片式列表。

### 3.1 首页结构

- [ ] 主操作突出：**创建任务** 按钮在首屏显著位置（与设计系统主按钮一致）。
- [ ] 标题与说明：首页主标题可保留一个，副标题或简短说明可选（如「上传招标文件，一键生成标书」），不重复堆砌。
- [ ] 任务列表区块：标题为「任务列表」或「我的任务」，与上方主操作有明确间距与层级。

**交付物**：  
- 首页结构清晰；主操作与列表分区明确。

### 3.2 任务列表展示

- [ ] 列表信息：至少包含状态、创建时间、操作（进入、删除）；「任务 ID」可弱化为次要信息或移至详情。
- [ ] 状态展示：用设计系统约定的状态标签/色块展示 pending / running / completed / failed 等，便于扫一眼识别。
- [ ] 行点击或「进入」链接：保持可进入任务详情；删除前二次确认保留。

**交付物**：  
- 任务列表信息与状态展示符合设计系统；进入与删除交互不变且清晰。

### 3.3 空态

- [ ] 无任务时：不只有表格的 emptyText，可增加插画或图标 + 一句引导文案（如「暂无任务，点击上方按钮创建第一个标书任务」）+ 主按钮或明显入口。
- [ ] 样式与设计系统一致（颜色、间距、字体层级）。

**验收标准**：  
- 首页主操作与列表层次清晰；列表状态易读；空态友好且符合设计系统。

## 阶段 4：设置页与对比页

**目标**：设置页分区清晰、状态提示明确；对比页布局与 diff 样式统一到设计系统。

### 4.1 设置页

- [ ] 分区：用卡片或区块区分「大模型 API」「模型配置」等；每区标题与间距统一。
- [ ] API 配置：每个 provider（DeepSeek、智谱等）一个子区块；已配置/未配置有明确状态（如「已配置」+ 脱敏 key）；输入框、Base URL、保存按钮布局清晰。
- [ ] 模型配置：默认模型与各步骤模型选择器排版清晰，与设计系统表单组件一致。

**交付物**：  
- 设置页分区与状态清晰；样式符合设计系统。

### 4.2 对比页

- [ ] 布局：两段文本输入 + 「对比」按钮 + diff 结果展示区；「从任务加载」入口与手动输入可 Tabs 或分区并存，不拥挤。
- [ ] Diff 展示：删除标红、新增标绿（或与设计系统语义色一致）；字体、行距、边距统一到设计系统；可选左右分栏或上下布局，与设计说明一致。
- [ ] 从任务加载：任务选择、类型（框架/章节）、章节号等控件排版清晰，加载中与错误态有反馈。

**验收标准**：  
- 设置页与对比页分区清晰、风格统一；diff 样式可维护且符合设计系统。

## 阶段 5：细节与打磨

**目标**：统一加载态、错误态、空态；兼顾响应式与可访问性；全站风格一致无遗漏。

### 5.1 加载、错误与空态

- [ ] **加载**：列表、详情、设置等数据加载时，使用统一的 Spin 或骨架屏样式（与设计系统一致）；按钮 loading 态统一。
- [ ] **错误**：接口失败或步骤失败时，错误信息展示位置与样式统一（如 Alert 或 Message）；「重试」等操作入口明确。
- [ ] **空态**：除首页任务列表外，其他列表或暂无数据处（如有）也提供简短说明与引导，风格与 3.3 一致。

**交付物**：  
- 加载/错误/空态在主要页面已统一；文案与样式可维护。

### 5.2 响应式与可访问性（可选）

- [ ] **响应式**：关键页面（首页、任务详情、设置）在常见宽度下（如 1280 / 768）布局不崩、主操作可点击；表格或长列表可横向滚动或适配。
- [ ] **可访问性**：按钮与链接有可读的文案或 aria-label；色彩对比满足基本可读性；键盘可聚焦关键操作。可不做完整 a11y 审计，但无明显障碍。

**交付物**：  
- 按需完成响应式与可访问性最小集；或明确留待后续迭代。

### 5.3 全站走查

- [ ] 逐页检查：首页、任务详情（含各步骤状态）、设置、对比；确认无遗漏的内联样式、未统一的颜色或字体、重复标题、开发用文案。
- [ ] 与设计系统文档或 token 对照：新加样式均使用 token 或主题变量，便于后续换肤或扩展。

**验收标准**：  
- 全站加载/错误/空态统一；响应式与可访问性按约定完成或记录为后续项；走查无明显风格与信息层级问题。

## 建议执行顺序与依赖

- **阶段 0** 必须最先完成，为后续所有页面提供 token 与主题基础。
- **阶段 1** 依赖 0（布局与导航会用到设计系统）；完成 1 后，所有页面已在统一布局下，便于逐页改版。
- **阶段 2** 依赖 0、1；任务详情页是核心，建议优先做，再做首页与列表。
- **阶段 3** 依赖 0、1；可与阶段 2 部分并行（如先做 2 再做 3，或 3 的列表与 2 的步骤条并行）。
- **阶段 4** 依赖 0、1；可与 2、3 并行或在其后。
- **阶段 5** 建议在 2、3、4 主要改版完成后做，作为收尾与全站统一。

每完成一个阶段，建议做一次小范围走查与验收，再进入下一阶段；若先出设计稿再开发，可将「交付物」拆为「设计交付」与「实现交付」两步，验收标准对应调整。

若你希望从某一阶段细化到「页面清单 + 组件清单 + 设计稿说明」，可以指定阶段号，再往下拆。
